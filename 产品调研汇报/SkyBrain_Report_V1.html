SkyBrain 产品规划与调研汇报

核心痛点分析

议题：分析 AI 编程工具在大型项目中的核心痛点

涵盖 "AI 生成代码质量" 与 "开发者使用体验" 两个维度

版本：V1 | 日期：2026-05-14

一、痛点总览

核心矛盾：AI 需要的上下文知识分散在人脑、文档、各仓库代码之中，缺乏一个统一的生命周期管理体系。

环节 回答的问题 核心高频痛点 次要 / 远期
知识生产 "AI 需要什么知识？谁来生产？" 8类 (A-H) 3类 (K, L, M)
知识分发与注入 "知识如何送达 AI？是否最新/精准？" 5类 (N, O, P, T, X) 1类 (W)
团队协作与持续进化 "团队如何协作使用 AI？如何持续改进？" 3类 (Q, R, S) 2类 (U, V)

经过深度调研与分析，共归纳出 13 类核心高频痛点、3 类次要低频痛点 和 3 类远期潜在痛点。

环节一：知识生产

上下文内容本身不存在或难以获取

核心矛盾：代码之外的领域知识、工程约定、架构决策等大量"隐性知识"未被系统化记录，AI 完全无法感知。

知识生产痛点 (1/4)

A. 跨模块调用失配

问题描述：大型项目按模块拆分后，AI 在单仓库编码时不了解被调用模块的接口契约、参数约定、返回值结构等，导致调用代码不匹配。

根因：跨仓库/跨模块的接口知识未被结构化记录。

B. 业务规则与隐式校验缺失

问题描述：复杂业务系统存在大量隐式规则（状态流转、字段联动、业务窗口期等），AI 无法感知，生成的代码遗漏关键校验。

根因：业务规则存在于代码之外（如PRD、流程图）。

知识生产痛点 (2/4)

C. 项目配置/约定缺失

问题描述：团队特有的编码规范（目录结构、命名规则、异常处理等）散落在文档或人脑中，AI 生成代码与规范不一致。

根因：团队编码规范无法被 AI 统一感知。

D. 历史遗留逻辑的"隐性知识"

问题描述：看似不合理的代码背后可能有业务原因或事故教训，AI 按常规最佳实践重构反而引入曾修复过的 Bug。

根因：无法获取代码变更历史、事故复盘等非代码信息。

知识生产痛点 (3/4)

E. 基础设施与中间件隐式约定

问题描述：在 MQ、缓存、DB 上的命名、路由等约定，AI 不了解，生成代码表面正确但部署后运行失败。

根因：基础设施层约定不在代码中可推导。

F. 第三方依赖封装层差异

问题描述：AI 绕过团队内部封装的 SDK（集中管理鉴权、日志、监控等），直接使用底层开源库，破坏统一管控。

根因：封装层的存在和用法通常在内部 SDK 文档中。

知识生产痛点 (4/4)

G. 新旧架构/API 并存选择困境

问题描述：架构演进中同一功能有新旧两套实现，AI 无法判断应该使用哪一套，经常混用或使用废弃方案。

根因：不了解项目的技术演进路线和迁移状态。

H. 外部工作流与系统孤岛

问题描述：AI 被“禁锢”在 IDE 中，无法读取 Jira(Bug)、Confluence(需求)、监控系统(报错) 等外部上下文，处于半盲状态。

根因：AI 编程工具与企业外部研发系统缺乏连接。

环节二：知识分发与注入

有了知识但无法正确、高效地送达 AI

核心矛盾：团队已有的上下文知识缺乏统一的分发机制，开发者需要手动配置、手动注入，且无法确保使用到的是最新版本。

知识分发与注入痛点 (1/2)

N. 上下文配置的重复劳动

问题描述：团队成员在 Cursor、Copilot、Trae 中分别配置 Rules，格式不互通，维护成本成倍增加。

根因：各 AI 工具格式不兼容，缺少统一分发标准。

O. 上下文保鲜困难

问题描述：代码和架构在演进，但本地上下文文档未更新，导致 AI 还在基于过时的上下文生成代码。

根因：上下文与代码脱钩，缺少"上下文版本管理"机制。

知识分发与注入痛点 (2/2)

P. 上下文注入靠人脑判断

描述：开发者手动挑选文档喂给 AI，判断错漏导致生成错误。缺少基于场景的智能推荐。

T. 上下文与代码版本不同步

描述：代码回滚或切分支时，上下文依然是最新的，AI 基于新架构生成旧分支代码导致不兼容。

X. 长对话"遗忘"退化

描述：多轮对话中，早期注入的规范被逐渐"挤出"上下文窗口，导致后期代码质量下降偏离约束。

环节三：团队协作与持续进化

团队层面的 AI 使用协作和持续改进机制缺失

核心矛盾：AI 编程目前是"个人工具"而非"团队工具"，缺乏团队层面的协作标准、反馈闭环和能力建设机制。

团队协作与持续进化痛点

Q. 团队 AI 使用能力参差不齐

描述：成员间 Prompt 和上下文配置能力差异极大，导致同样任务产出质量天差地别。
根因：AI 能力是"个人资产"而非"团队资产"，缺少标准化传承。

R. AI 产出的代码格式/风格不一致

描述：不同工具或个人配置导致生成的代码风格迥异（如 Stream vs For），增加 CR 成本。
根因：团队级别的代码风格偏好没有统一注入到各 AI 工具中。

S. 缺乏团队级 AI 使用的反馈闭环

描述：开发者手动修复 AI 的错误后，经验只留存在个人记忆中，AI 下次依然"重蹈覆辙"。
根因：缺少从"AI 错误"到"上下文改进"的团队反馈机制。

四、痛点全景图与解决方向

环节 核心矛盾 解决方向
知识生产 隐性知识未被系统化记录 建立结构化采集和外化机制
知识分发与注入 缺少统一分发标准，依赖人工 建立统一协议和智能注入机制
协作与持续进化 缺团队标准和反馈闭环 建立团队级使用标准和反馈闭环

关键洞察 (Key Insights)

  • 环环相扣：知识生产是根基，分发与注入是管道，协作与进化是土壤。
  • 痛点叠加：如"跨模块调用失配"叠加"封装层差异"会导致 AI 完全失去方向。
  • 演进路线：SkyBrain 的 MVP 应优先覆盖环节一(生产)和环节二(分发)，远期痛点（可观测性、跨工具兼容）需预留架构空间。

章节二：数据如何存储

SkyBrain 知识体系的存储与流转架构

涵盖知识形态、分级投递、作用域、生命周期及物理存储

1. 知识的形态划分 (四大类九种)

解答：“SkyBrain 到底存什么？”

类别 作用 具体形态 (9种)
📋 操作型 告诉 AI "怎么做" Skill(场景范例)、Workflow(业务流程)、Reference(依赖用法)
📐 契约型 告诉 AI "是什么" API Spec(接口契约)、Schema(表结构模型)
📖 认知型 告诉 AI "项目大局" Architecture(架构概述)、Glossary(业务术语)
⚖️ 约束型 告诉 AI "红线在哪" Rule(硬性安全技术约束)、Convention(编码风格约定)

2. 知识的分级与双通道投递架构

基于知识特征推导：“什么时候出现，通过什么通道喂给 AI？”

L1 元知识 & L2 固化知识

特性：必须 100% 送达，AI 不能跳过（包含 SkyBrain 使用说明、全局架构与约束）。

通道：磁盘写入 (强制注入)

渲染为 .cursorrules 等静态文件，注入 System Prompt。

L3 动态扩展知识

特性：长尾、庞大、专业（API、Skill 等），按需检索，防止 Token 撑爆。

通道：MCP 动态查询 (按需拉取)

AI 受 L1 元指令驱动，主动向 Local Daemon 发起查询。

3. 知识的四级作用域与继承分发

解决痛点：避免上下文爆炸，实现知识的“精准滴灌”

核心理念：知识的作用域(空间)与版本演进(时间)解耦，采用下层覆盖上层的树状继承模型。

  • L1 Org (公司级)：全公司/跨项目通用的底层规范（如 Java 编码规约）。
  • L2 Project (项目级)：项目概述、整体架构、核心技术栈。
  • L3 Module (模块级)：根据当前编辑文件路径嗅探，投递特定模块的接口或状态机逻辑。
  • L4 Branch (分支级)：根据 Git HEAD 嗅探，投递仅针对当前特性的临时知识。

4. 知识生命周期衰减与垃圾回收 (GC)

解决痛点：知识腐化导致 AI 产生“幻觉”并引发大量返工

双通道反馈采集

  • MCP 会话内即时反馈：在 VibeCoding 期间自然语言反馈或抽样轻量评价。
  • Daemon 面板手动标注：事后按知识条目进行 👍/👎 一键标注。

评分引擎与处理策略

  • HealthScore：手动评价(主力) + 时间/频次衰减(兜底)。
  • 逻辑删除：分数低于阈值，知识对 AI 隐藏（移出检索），但保留归档以供追溯或复活。

5. 知识的物理存储架构方案

当前上下文知识库常见存储方案的综合选型：

存储层 技术选型建议 存储内容与作用
结构化元数据 PostgreSQL / MySQL 存储知识条目的基本信息、作用域标签(Org/Proj/Mod)、版本号、作者及 HealthScore 评分。
向量检索库 Milvus / Qdrant / Pinecone 对操作型(Skill/Workflow)等动态知识进行 Embedding 化，支持 AI 意图触发时的高效语义相似度匹配。
文本/内容对象 S3 对象存储 / MongoDB(文档) 存储具体的长文本、代码范例 (Markdown 格式)，应对可能较大的知识体量。
本地缓存层 SQLite / 本地文件 (Daemon 端) Local Daemon 本地缓存 L1/L2 静态规则，确保离线可用和极速的 System Prompt 渲染。

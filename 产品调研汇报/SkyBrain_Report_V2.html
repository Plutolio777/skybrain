SkyBrain

企业级 AI 编码上下文引擎与存储架构方案

产品调研汇报 V2.0

第一篇章：核心痛点 —— 知识生产的“真空期”

业务与规范的“黑盒”：代码之外的隐性知识未被系统化记录，导致 AI 编码时错漏百出。

跨模块调用失配 & 封装层差异

**场景**：编辑 `apt_rules` 时需依赖 `xenadmin`；团队有内部HTTP SDK封装。
**灾难**：AI缺乏跨仓库知识，基于错误签名生成代码；裸调开源库，绕过团队监控与安全审计。

基础设施约定缺失

**场景**：团队有统一的 Redis key 命名风格以及数据库表命名风格。
**灾难**：AI 乱起 Key 名和表名，完全无视团队的底层数据架构规范，导致后期难以维护清理。

业务规则与隐式校验缺失

**场景**：在编写 `xenadmin` 告警相关代码时，不清楚底层告警数据流转。
**灾难**：规则仅在少数人脑中，AI 生成代码漏掉流转校验，直接导致线上故障或严重资损。

历史遗留逻辑被“重构”

**场景**：代码中某段看似冗余的校验，实际是历史事故后加的防御性检查。
**灾难**：AI按“最佳实践”自作聪明地删除了防御校验，导致老 Bug 秽土重生。

新旧架构并存 & 外部工作流孤岛

**场景**：项目新旧接口共存；修复 Bug 时需对接 Jira 获取需求与缺陷上下文。
**灾难**：AI 混用废弃方案；被禁锢在 IDE 中无法拉取外部上下文，开发者沦为复制粘贴的“人肉搬运工”。

👉 引出核心拷问一：【知识怎么上传】 开发者极度抗拒写文档，面对无文档的祖传老项目，如何做到“零摩擦”的沉淀与冷启动？

第一篇章：核心痛点 —— 协作与注入的混乱

AI 仍是“个人玩具”，团队缺乏底层管理，且知识注入完全依赖人脑，管道严重堵塞。

产出风格撕裂 & 配置重复劳动

**场景**：协作开发微服务；团队有统一规范。
**灾难**：A的AI用Stream，B用for循环；要在Cursor、Copilot中手写多份不兼容副本，维护成本极高，Review沦为纠偏灾难。

缺乏反馈闭环 & 版本不同步

**场景**：AI 总是搞错某接口参数；代码回滚到旧分支。
**灾难**：A修了错误但经验只留脑中，B遇同问题AI继续犯错；本地Rules未更新，或知识库最新但代码旧，AI生成代码直接编译报错。

注入靠人脑 & 长对话“遗忘”

**场景**：修改接口需遵循约定；或进行10轮以上的复杂重构。
**灾难**：全凭记忆找文档喂给AI，漏注则出错；早期注入的规范被“挤出”窗口，AI 后期突然乱抛异常。

👉 引出核心拷问二与三：【怎么使用 & 怎么存储】
窗口宝贵，如何恰当喂给 AI？面对海量数据与各怀鬼胎的个人配置，逻辑上如何防止被错误规则污染全局大脑？

第二篇章：破局之道 —— SkyBrain 上下文数据库引擎

要解决上述痛点，回应“存、用、传”三大拷问，我们需要彻底重构 AI 编码的基础设施：

不仅仅是插件，而是“数据库引擎”

SkyBrain 的核心定位不是一个简单的 IDE 规则生成器，而是一个贯穿知识全生命周期的管理枢纽：

  • A. 解决“怎么存” 建立明确的内容形态分类、防污染生命周期与多级作用域继承机制。
  • B. 解决“怎么用” 建立 L1/L2/L3 双轨制分级加载体系，静态强制注入与动态 MCP 检索结合。
  • C. 解决“怎么传” 引入 MCP 对话式零摩擦写入，与基于 Agent 的代码库逆向提炼冷启动。

第二篇章：破局之道 —— 第一步：搞清“存什么”

产品设计的起点必须是内容形态。只有先搞清楚“知识长什么样”，才能推导后续体系。

📋 操作型（告诉 AI “怎么做”）

面向特定任务的执行指南，只有 AI 准备做同类事情时才需要。

  • **Skill**：特定场景操作范例
  • **Workflow**：跨模块业务流
  • **Reference**：外部依赖用法

📐 契约型（告诉 AI “是什么”）

定义系统的接口边界和数据结构，与特定模块强绑定。

  • **API Spec**：接口签名与契约
  • **Schema**：数据表/实体定义

📖 认知型（告诉 AI “在哪、是什么”）

理解整个项目的前提，缺失会导致 AI 从根源上犯错。

  • **Architecture**：整体架构与分层设计
  • **Glossary**：业务术语定义

⚖️ 约束型（告诉 AI “绝不能做什么”）

团队的红线与公约，违反会引发生产事故或代码风格混乱。

  • **Rule（红线）**：硬性安全/技术约束
  • **Convention（风格）**：编码/命名约定
通过将混沌的知识结构化为这四大类九种形态，我们彻底剥离了“内容”与“投递通道”的耦合，为后续的按需加载打下坚实基础。

第二篇章：破局之道 —— 第二步：从内容推导生效范围

搞清楚了“存什么”，自然就能推导出“什么时候需要它”以及“怎么喂给 AI”。

推导链路：形态 → 生效范围 → L1/L2/L3 加载体系

  • **约束型 + 认知型**（红线与常识）：
    必须 **全局常驻**。因此推导出走 **L1/L2 磁盘写入通道**，渲染为静态规则强制注入，确保 100% 送达。
  • **契约型**（接口与表结构）：
    属于 **模块绑定**。仅在编辑特定模块时需要，跨模块注入纯属噪音。因此推导出走 **L3 MCP 动态查询**。
  • **操作型**（Skill 与工作流）：
    属于 **意图触发**。只有 AI 识别到相关任务（如“写支付逻辑”）时才需要查阅范例。推导出走 **L3 MCP 动态查询**，按需加载，绝不撑爆上下文。

第三篇章：行业前沿调研 —— 发展全景

在解决“知识怎么存”之前，我们先看当前 AI 知识库领域的发展现状与核心悖论。

行业现状：高采纳，低转化

  • **71%** 的组织定期使用 GenAI，但仅 **17%** 能产生可衡量的可观利润。
  • **40-60%** 的 RAG 项目无法进入生产环境。
  • **95%** 的企业 GenAI 投资未产生可量化影响。

根本原因剖析

大多数企业只是“给现有系统打 AI 补丁”，而非以 AI 为底层逻辑**重构知识基础设施**。用传统的思维做 AI 知识库，注定烂尾。

行业前沿调研：三大主流架构流派

1. 向量式 RAG（2019-2025 主流）

**架构**：切片(Chunking) → 向量化 → 相似度检索。
**缺陷**：切片切断了跨文档的关联（无法多跳推理）；对“全局主题”无能为力；检索失败率高。

2. GraphRAG（2025-2026 爆发）

**架构**：多层次知识表示（向量 + 知识图谱 + 层次索引）。
**创新**：Leiden 社区检测，从底层向上逐层生成摘要。双模式查询（Local/Global）。
**预测**：到 2026 年 85% 的企业将采用向量+图谱混合架构。

3. Karpathy 结构型 Wiki（2026 新兴）

**架构**：LLM 主动编译 Markdown Wiki → 直接全量注入长上下文。
**创新**：不需要向量库，LLM 充当“全天候研究员”持续保鲜。知识编译一次，而非每次重新检索碎片。

行业前沿调研：2026-2030 关键趋势与映射

五大关键趋势

  • 从检索管线演化为**自治知识运行时**。
  • **多模态知识表示**（向量+图谱+索引）成标配。
  • **MCP** 成为 AI 与知识库的连接标准。
  • **知识治理**（防污染/GC）成为一等公民。
  • **超长上下文**：全量喂入 vs 检索推理精准性。

对 SkyBrain 的启示

  • **GraphRAG 社区检测** 映射我们的四级作用域。
  • **Karpathy Wiki 模式** 映射我们的三级生命周期（AI辅助编译的团队版）。
  • **MCP 标准** 坚定了 Local Daemon 架构。
  • **知识治理** 对应我们的 HealthScore GC 机制。

第四篇章：SkyBrain 架构落地 —— 节点 A：知识怎么存

物理维度：务实的混合存储架构

  • **PostgreSQL**：承载树状目录、四级作用域关联、点赞评分等元数据，支撑复杂的权限与生命周期流转。
  • **Qdrant**：专门处理高维向量，提供极速的语义相似度检索。

逻辑维度：四大内容形态分类

  • **操作型**：Skill、Workflow、Reference。
  • **契约型**：API Spec、Schema。
  • **认知型**：Architecture、Glossary。
  • **约束型**：Rule/Convention。

生命周期维度：防污染治理

建立个人沙盒 → 团队提案 → 中央大脑的三级准入控制，并按 Org/Project/Module/Branch 划分作用域。

SkyBrain 架构落地 —— 节点 B：知识怎么用

由知识形态推导出的加载体系，解决单纯依赖 MCP（可能偷懒不查）和静态文件（Token爆炸）的矛盾。

L1/L2/L3 双轨制分级加载体系

  • **轨道一：磁盘写入强制注入 (L1 + L2)**
    L1 元知识（强制 MCP 查询指令）与 L2 固化知识（红线规约/认知常识）。通过 Local Daemon 渲染为 `.cursorrules` 等静态文件强制注入，AI 无权拒绝。
  • **轨道二：MCP 动态查询按需拉取 (L3)**
    L3 动态扩展知识（操作型 Workflow、契约型 API 等长文本）。由 L1 元指令驱动 AI 遇到任务时主动通过 MCP 获取，解决窗口遗忘。

多 IDE / 多端优雅适配

云端维护统一 Markdown 知识，Local Daemon 根据环境探针（Cursor/Trae）翻译降级，实现“一处维护，多端生效”。

SkyBrain 架构落地 —— 节点 C：知识怎么传

MCP 对话式反向写入 (零摩擦录入)

颠覆 MCP 仅作“查询”的习惯，打通写入链路：

  • **自然语言触发**：开发者顺手一句：“把这段逻辑作为规范保存到 SkyBrain”。
  • **自动提取入库**：AI 识别意图，调用 `save_to_skybrain`，直接存入个人沙盒，不打断编码心流。

Agent 代码库自动提炼 (冷启动)

针对无文档存量项目的“知识真空期”：

  • **多维探针静默扫描**：部署 AST 探针、配置探针、目录结构探针。
  • **结构化逆向工程**：Agent 根据“四大类九种形态”模板，自动提取隐藏规约和接口协议，生成知识提案，快速完成初建。

第五篇章：支撑体系与演进路线

支撑体系：双端形态与 GC 衰减

  • **Web 端控制台**：面向管理者，负责全局资产管理、团队提案审核（Review）与大屏度量。
  • **Local Daemon**：面向开发者的本地伴生沙盒。
  • **GC 衰减机制**：HealthScore 算法 = 手动评分(高权) + 时间/频次衰减，防知识腐化。

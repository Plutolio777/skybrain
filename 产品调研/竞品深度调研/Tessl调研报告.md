# 竞品深度调研：Tessl

> 调研日期：2026-05-14
> 产品官网：[https://tessl.io/](https://tessl.io/)
> 文档地址：[https://docs.tessl.io/](https://docs.tessl.io/)
> 核心定位：AI 原生开发的“意图与规范”平台 / Agent 技能的包管理器

---

## 一、 产品愿景与定位

Tessl 的核心理念是推动软件开发范式的转移：**从“手写代码”转向“编写意图与规范（Spec-driven development）”**。

在企业级 AI 编程的语境下，Tessl 定位为一个**“Agent 技能与上下文的包管理器”**。它认为痛点不在于大模型本身的能力，而在于企业如何集中管理、版本化、并向各种 AI 编程助手（Claude Code, Cursor, Copilot 等）分发项目的架构约定与领域知识。

---

## 二、 核心概念与机制

Tessl 的架构围绕三个核心概念展开，这与我们在 V2 前置议题中探讨的“知识多模态（Rule/Skill/Workflow）”高度吻合：

### 1. Specs（规范 / 意图描述）
*   **定义**：Specs 是 Tessl 系统的**单一事实来源（Single Source of Truth）**。开发者使用结构化的自然语言（辅以特定语法）来描述系统的需求、接口定义或业务逻辑。
*   **生命周期**：Specs 像代码一样支持版本控制。当业务需求变更时，开发者只需修改 Spec，Tessl 就能感知到变更，并驱动底层的 AI 代理去更新具体的代码实现。
*   **对 SkyBrain 的启发**：这解决了“业务规则隐式存在于人脑”的问题，将非结构化需求转化为 AI 可直接消费的规范资产。

### 2. Skills（技能库）
*   **定义**：赋予 AI 执行特定专业任务的“能力插件”。例如“如何连接公司内部的鉴权服务”、“如何生成符合特定框架规范的 DTO”。
*   **管理机制**：Tessl 允许团队像管理 npm 包一样去沉淀、版本化这些 Skills。
*   **对 SkyBrain 的启发**：验证了“能力型知识”必须独立封装并可分发的必要性，解决了“团队 AI 能力参差不齐”的问题。

### 3. Workflows（自动化编排引擎）
*   **定义**：连接 Specs、Skills 与底层代码的流水线。
*   **运行机制**：基于触发器运行。例如：当一个 Spec 被修改后，Workflow 会自动触发 AI 生成业务代码 -> 运行单元测试 -> 校验安全规范 -> 自动创建 PR。
*   **对 SkyBrain 的启发**：AI 编程不仅仅是编辑器里的“自动补全”，更应该与 CI/CD 结合，形成闭环的工作流。

---

## 三、 IDE 集成与上下文分发

Tessl 在解决“IDE 孤岛”和“上下文遗忘”方面采取了以下策略：

1.  **无缝的 IDE 插件**：Tessl 提供主流 IDE（如 VS Code）的官方插件。开发者不需要跳出编辑器去网页端，可以直接在 IDE 面板中编写 Specs、审查 AI 生成的代码。
2.  **动态上下文注入**：系统会根据开发者当前打开的文件或正在执行的任务，**按需精准召回**对应的 Specs 和 Skills 注入给底层模型。这有效对抗了大模型的上下文窗口超载和长对话遗忘问题。
3.  **跨工具适配枢纽**：Tessl 本身不绑定某一款特定模型或 IDE。它向下屏蔽各 IDE 的差异，将团队集中的知识规范，动态转换为 Cursor、Claude Code 等不同 AI 工具能够理解的底层指令。

## 四、 优劣势分析（SWOT）

### 优势 (Strengths)
*   **理念前卫**：真正践行了“AI 编写代码，人类编写规范”的未来开发范式。
*   **生命周期管理**：将零散的 Prompt 升级为了支持版本控制、依赖管理的工程资产（Package Manager 理念）。
*   **生态开放**：Agent-agnostic（对 Agent 无关），能兼容多种主流 AI 编码工具。

### 劣势 (Weaknesses)
*   **实施门槛高**：要求开发团队彻底改变工作习惯，从“写代码”变为“写高质量的 Specs”，这在短期内对开发者的抽象建模能力要求极高。
*   **重构老项目困难**：对于已经积累了大量技术债、缺乏文档的存量企业级项目，很难低成本地“冷启动”生成全套 Specs。

---

## 五、 对 SkyBrain 的战略启示

1.  **产品形态的验证**：Tessl 的成功证明了**“独立后台集中管理 + IDE 轻量插件分发”**的架构（我们在议题一中构想的形态 C/B）是企业级市场的最优解。
2.  **概念的融合**：SkyBrain 必须吸收 Tessl 将“Rule / Skill / Workflow”作为一级实体进行生命周期管理（创建、版本化、分发、度量）的理念。
3.  **我们的差异化切入点（避开 Tessl 的坑）**：
    *   Tessl 侧重于**“自上而下”**（先写 Spec，再生成代码），这适合新项目。
    *   SkyBrain 应该强化**“自下而上”**的知识抽取能力（从现有的 Jira 任务、报错日志、旧代码中自动反向提炼上下文），以此来降低企业老项目的冷启动门槛。

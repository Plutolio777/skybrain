# SkyBrain 解决方案 V2 — 脑洞探索：代码库自动扫描与冷启动构建

> 状态：讨论完成（已提取）
> 阶段：V2 解决方案与产品雏形
> 核心问题：SkyBrain 作为一个团队大脑，在刚刚接入一个具有大量历史代码的存量项目时，面临严重的"知识真空期"。如何不依赖人工录入，通过自动化手段快速建立初始上下文？

---

## 一、 问题缘起：冷启动的"知识真空期"

尽管我们在前面的设计中规划了 MCP 对话式录入和三级生命周期（沙盒->提案->中央大脑）来确保知识的持续积累。但这依赖于开发者长期的使用和沉淀。

当团队将 SkyBrain 接入一个已有 50 万行代码的存量项目的第一天，中央大脑里关于这个项目的知识是 **0**。
在这个"真空期"内：
1. 开发者在 IDE 中使用 AI 编写代码时，AI 缺乏项目背景（不知道用了什么框架、有什么既定规范）。
2. AI 会持续生成风格不一致、依赖错误的代码，导致开发者体验极差。
3. 如果要求 TL 纯手工去梳理所有历史知识，这会带来巨大的迁移成本，直接导致产品落地失败。

因此，我们需要一种**冷启动机制**，让 SkyBrain 在接入瞬间就能"略懂"这个项目。

---

## 二、 核心设计理念：代码库即真相 (Codebase as the Source of Truth)

最真实的规范和知识，往往就隐藏在现有的代码中。
**设计理念**：利用 Local Daemon 本地算力和大模型的代码理解能力，在项目接入初期，对代码库进行一次**深度静默扫描**，逆向工程提取出项目的"基线知识"（Baseline Knowledge）。

这些由 AI 自动扫描提取的知识，不会直接进入中央大脑的"真理区"，而是进入**提案区（Team Proposal）**，等待 TL 审核后正式生效。

---

## 三、 最终方案：自动扫描与提炼流水线

### 1. 触发时机与扫描模式
- **初次接入触发**：当 Local Daemon 首次在某个项目目录下被激活，且发现云端该项目知识库为空时，自动弹出提示："检测到新项目，是否启动智能冷启动扫描？"
- **静默后台执行**：扫描过程利用开发者本地算力（或指定的构建服务器）在后台执行，避免占用正常开发的带宽和性能。

### 2. 探针设计（知识提取目标）
扫描引擎会部署不同维度的"探针"，针对特定文件进行特征提取：

| 探针类型 | 目标文件特征 | 提取出的知识类型（示例） |
| --- | --- | --- |
| **依赖探针** | `package.json`, `pom.xml`, `go.mod` | 基础技术栈（如：本项目使用 React 18 + TailwindCSS；禁止使用 Moment.js，已替换为 dayjs） |
| **配置探针** | `tsconfig.json`, `.eslintrc`, `application.yml` | 编译标准、代码风格约束、多环境配置策略 |
| **结构探针** | 目录树扫描 (`src/`, `components/` 等) | 架构分层规范（如：`src/api` 统一存放接口请求，`src/components` 存放无状态组件） |
| **文档探针** | `README.md`, `CONTRIBUTING.md` | 项目整体概述、核心业务术语、启动与部署命令 |
| **AST 模式探针** | 核心业务代码的抽象语法树分析 | 错误处理范式（如：统一抛出 `BusinessException`）、日志打印规范 |

### 3. 知识提炼与转化 (Raw Data -> Knowledge Rules)
提取出的原始信息会经过 LLM 的二次提炼，转化为符合 SkyBrain 标准格式的知识规则。
- *原始信息*：扫到项目中所有的 API 请求都封装在一个 `axios_instance.js` 中。
- *转化知识*：`【网络请求规范】所有前端网络请求禁止直接使用原生 fetch 或 axios，必须导入 @/utils/axios_instance。`

### 4. 审核与接纳（冷启动安全阀）
为了防止自动扫描产生"幻觉"（比如把一段废弃的烂代码当成了规范）：
1. 扫描生成的所有知识，其初始状态全部标记为 `Draft`（草案）。
2. 这些草案知识打包成一个**"冷启动知识提案"（Cold-start Proposal）**，推送到 Web Console 的 TL 审核看板。
3. TL 可以批量浏览、修改或拒绝这些规则。
4. 审批通过的部分，正式打上 `Project` 级别的作用域标签，进入中央大脑。

---

## 四、 挑战与应对

### 1. 巨型单体仓库（Monorepo）的扫描耗时问题
- **挑战**：几百万行代码的项目，全量扫描会消耗大量 Token 和时间。
- **应对**：引入**渐进式扫描（Progressive Scan）**。首日仅扫描核心配置文件（Package、Config）和目录结构；后续开发者访问到具体模块（如 `apps/admin-panel`）时，再触发该模块局部的深入扫描（懒加载模式）。

### 2. 代码库中存在多种不一致的风格
- **挑战**：老代码用 A 风格，新代码用 B 风格，AI 扫描该信谁？
- **应对**：基于 Git Blame 信息，给予**近期活跃代码**更高的权重。AI 提炼时若发现冲突，可在提案中并列展示，并在备注中标注："发现两种不同的错误处理模式，建议 TL 确认最终规范"。

---

## 五、 结论与后续行动

通过"代码库自动扫描冷启动"，SkyBrain 将原本需要数周的人工梳理过程压缩到了几个小时。它不仅解决了新项目接入的"知识真空"痛点，更是对团队现有隐性知识的一次系统性盘点。

结合**"四级作用域分发体系"**，这套冷启动扫描出来的知识，可以被精准地分配到 Project 或 Module 级别，为开发者提供一个坚实、可靠、无缝介入的 AI 编程基座。

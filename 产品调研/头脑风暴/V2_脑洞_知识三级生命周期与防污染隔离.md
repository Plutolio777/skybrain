# SkyBrain 解决方案 V2 — 脑洞探索：知识三级生命周期与防污染隔离机制

> 状态：讨论完成
> 阶段：V2 解决方案与产品雏形
> 核心问题：极简录入必然带来垃圾输入。如果任何开发者都能随手把规则写进中央大脑，如何防止团队上下文被低质量知识污染？

---

## 一、 问题缘起：无摩擦录入的"黑暗面"

在前一个脑洞中，我们确立了 MCP 对话式反向写入——开发者在 AI 对话框里顺嘴说一句就能完成知识沉淀。这个"零摩擦力"入口威力巨大，但也带来了一个致命风险：

> 如果每个开发者的随笔、个人偏好、未经验证的"我觉得应该这样写"，都能直接进入中央大脑并喂给全团队的 AI——那么 SkyBrain 会从"知识引擎"迅速退化为"谣言扩散器"。

**类比 Git 的教训**：Git 之所以能支撑大规模协作，不是因为"谁都能直接推到 main"，而是因为有 `fork → PR → review → merge` 这一整套质量控制链。SkyBrain 的知识库同样需要这种"准入控制"。

---

## 二、 核心设计理念：Git PR 流的知识管理版本

经过讨论，我们为 SkyBrain 设计了类似 Git PR 流的 **"知识三级生命周期隔离与审核机制"**：

1. **个人沙盒层**（类似 Git 的本地 branch）——隔离验证区
2. **团队提案层**（类似 Git 的 PR）——云端待审核
3. **中央大脑层**（类似 Git 的 main）——全局真理，全员生效

每一层有严格的准入条件和流转规则，知识从"个人想法"到"团队共识"需要经过两道关卡。

---

## 三、 最终方案：三级生命周期详解

### 层级一：个人沙盒层 (Personal Sandbox) —— "隔离验证区"

- **触发**：所有通过 MCP `save_to_skybrain` 写入的知识，默认进入此层
- **物理隔离**：知识存储在开发者本机的 Local Daemon 数据库中，绝不上传云端
- **可见性**：仅对自己可见。第二天开发者提问时，Local Daemon 会合并检索"云端大脑"和"本地沙盒"，开发者能享受自己沉淀的知识，但不影响任何人
- **生命周期**：开发者可以随时编辑、删除、升级到提案层
- **设计意图**：给开发者一个"安全试错空间"——随便写、随便改，不会污染任何人。就像 Git 的本地分支，在自己电脑上怎么折腾都行

### 层级二：团队提案层 (Team Proposal) —— "云端待审核"

- **流转机制**：开发者在个人沙盒中验证某条知识有效后，通过 MCP 指令"把这条安全鉴权规则提报给团队"
- **物理流转**：Local Daemon 将数据打包，通过 API 上传至云端 Web 控制台
- **AI 预审（Gateway）**：
  - 云端后台的大模型对提案进行自动冲突检测
  - 如果提案与现有全局规范存在硬冲突 → 直接打回给提交者，附说明
  - 若无硬伤 → 进入 `Proposal` 状态，排队等待人工审核
- **设计意图**：先用 AI 做第一轮过滤，减少 TL 的审核负担。AI 不需要做最终判断，只需要识别"明确的硬冲突"

### 层级三：中央大脑层 (Global Brain) —— "全局真理"

- **审核机制**：TL 或架构师登录 Web 端控制台，在"待审核面板"中看到提案
- **审批操作**：
  - ✅ Approve → 知识正式并入中央大脑，对全团队 MCP Server 可见，全员生效
  - ❌ Reject → 退回提交者，附原因
  - 🔄 Request Changes → 要求修改后重新提交
- **设计意图**：最终审批权在"人的判断"——AI 可以辅助但无权替代。这是保持知识库质量的最后一道防线

## 四、 方案面临的核心挑战与应对策略

### 挑战 1：TL 审批瓶颈

**问题**：如果团队每天有几十条提案涌入，TL 一个人审不过来，审核就变成阻塞点。

**应对**：
- AI 预审承担第一轮过滤，打回明确有问题/冲突的提案
- 支持"知识域（Domain）负责人"分级审核——支付模块的规则由支付 TL 审，安全规范由安全负责人审，而非全部压在一个人身上
- 优先级排序：高频使用模块的提案排前面，冷门模块可推迟

### 挑战 2：沙盒里的好东西永远不出来

**问题**：开发者在自己沙盒里沉淀了很多高质量知识，但"提报的心理门槛"太高——他们会想"这条是不是还不够完美？要不要再打磨一下？"结果永远在沙盒里沉睡。

**应对**：
- Local Daemon 定期检测沙盒中"被高频使用但尚未提报"的知识，主动提示开发者"这条知识你已经用了 15 次了，效果看起来不错，要不要提报给团队？"
- 降低提报的心理门槛：明确告知"提案会被 AI 预审，有问题会打回你能改，不是最终判决"

### 挑战 3：层级之间的知识版本同步

**问题**：开发者在沙盒里改了知识 → 提报 → TL 审过了 → 进入中央大脑。但原开发者后来又改了沙盒里的版本，两边不一致怎么办？

**应对**：
- 知识一旦提报（Promote），沙盒中的原版自动标记为"已提报副本"，不再允许编辑
- 如果需要修改已进入中央大脑的知识，需发起新的 "Update Proposal"，走同样的审核流程
- 中央大脑保留知识的版本历史，支持 diff 对比

## 六、 结论与后续行动

本次脑洞探索为 SkyBrain 建立了一套"去中心化生产 + 集中式把控"的护城河：

1. **三层隔离**：个人沙盒 → 团队提案 → 中央大脑，层层设卡，质量递进
2. **Git PR 范式迁移**：将软件工程中久经考验的代码审查流程，映射到知识管理领域
3. **AI 预审 + 人工终审**：AI 做第一轮过滤（查冲突、查重复），人工（TL）做最终判断，各司其职
4. **与知识 GC 联动**：将来知识 GC 判定某条知识过期后，不是直接删除，而是"降级回提案区"——本质上就是三级生命周期的逆向流转

这套机制是整个 SkyBrain 知识质量体系的**骨架**。没有它，MCP 无摩擦录入就是一个"开源水库"——人人可倒，无人过滤，最终变成垃圾堆。

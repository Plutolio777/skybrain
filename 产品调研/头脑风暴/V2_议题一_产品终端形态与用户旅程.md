# SkyBrain 解决方案 V2 — 议题一：产品终端形态与交互架构

> 状态：已完成讨论
> 阶段：V2 解决方案与产品雏形
> 核心议题：SkyBrain 在开发者端究竟长什么样？知识如何与现有的 IDE 工具链（Cursor, Trae 等）发生交互？

---

## 一、 产品宏观形态：云端实体 + 本地双轨制客户端

经过激烈的讨论与碰撞，我们推翻了单一形态的设想，确立了 SkyBrain 的宏观产品形态为 **“Web 端实体 + 本地无感驻留客户端”** 的双端协同模式。

### 1. Web 端控制台（实体存在）
**定位：宏观管理与全局视野**
*   系统必须存在一个可视化的实体（Web 后台），以满足重度操作需求。
*   **核心功能**：全局配置、外部系统（Jira/GitLab）授权绑定、大部头知识文档（PDF/长文）上传、团队知识图谱可视化、以及全局规范的 Knowledge Review（审核）流。

### 2. 本地守护客户端（IDE 伴生管家）
**定位：无感桥接与上下文投递**
*   对于一线开发者，避免强制安装笨重的界面客户端。SkyBrain 在开发者电脑上表现为一个**轻量级的托盘程序/后台守护进程（Daemon）**。
*   它不提供复杂的 UI，而是作为 Web 端与本地 IDE（Cursor/Trae）之间的“翻译官”和“搬运工”。

---

## 二、 核心架构突破：双轨制上下文投递（L1 + L2）

在探讨如何将知识喂给大模型时，我们发现单纯依赖 MCP（Model Context Protocol）存在致命缺陷——MCP 依赖大模型主动调用，对于“绝不能违背的基础规范”，大模型可能会因为“过度自信”而拒绝查询。

因此，SkyBrain 在本地客户端与 IDE 的交互上，确立了 **“静态强制注入（L1） + 动态渐进拉取（L2）”的双轨制架构**：

### 轨道一：L1 静态强制注入（潜意识/常驻内存）
*   **适用内容**：固定知识、项目概述、模块划分字典、不可违背的安全/开发规范。以及极其重要的一句话：“当你需要特定技能时，必须调用 MCP 知识库”。
*   **工作机制（同步器）**：SkyBrain 本地客户端充当**“规则同步器”**。它会从云端拉取当前项目的 L1 核心知识，并在后台静默地将其渲染为目标 IDE 认识的静态文件（例如在 Cursor 项目根目录自动生成并更新 `.cursorrules` 文件）。
*   **价值**：利用 IDE 自带的机制，**强制**将这部分知识作为 System Prompt 注入，大模型无权拒绝，必须遵守。

### 轨道二：L2 动态渐进获取（外存/按需工具箱）
*   **适用内容**：长篇业务流（Workflows）、具体的 API 定义、特定 SDK 的调用范例（Skills）、数据库表结构等“长尾且庞大”的知识。
*   **工作机制（MCP Server）**：SkyBrain 本地客户端同时作为一个标准的 **MCP Server** 运行。它向 IDE 暴露 `search_skybrain_skills()`、`get_api_spec()` 等标准工具。
*   **价值**：大模型在 L1 的强制指导下，遇到具体任务时，会**主动**通过 MCP 接口向 SkyBrain 客户端拉取这些长篇细节。完美解决了长对话遗忘和上下文窗口爆炸的痛点。

## 三、 开发者一天的 User Journey（场景推演）

1.  **早晨开启环境**：开发者小明打开电脑，SkyBrain 后台客户端自动启动，静默拉取了公司最新发布的《V2 支付服务重构规范》，并自动更新了小明本地项目的 `.cursorrules`（L1 轨道更新完成）。
2.  **接需求写代码**：小明在 Cursor 中要求 AI 开发一个支付回调接口。AI 读取到 `.cursorrules` 中的要求：“开发接口必须查询 SkyBrain 的 `payment_api_skill`”。
3.  **动态拉取知识**：AI 自动通过 MCP 协议（L2 轨道）调用 SkyBrain 本地服务，拉取到了最新的回调鉴权代码范例。
4.  **高质量输出**：AI 结合 L1 的基础规范和 L2 的鉴权范例，一次性生成了完全符合团队标准的代码，小明无需进行任何上下文的复制粘贴。

---

## 四、 结论与意义

议题一的讨论确立了 SkyBrain 最核心的运转逻辑：**不改变开发者的使用习惯，不依赖大模型的自觉性，通过“云端管理 + 本地双轨制投递”建立了一套坚不可摧的上下文工程底座。** 

这为后续我们探讨“知识应该如何录入（议题二）”和“如何进行图谱化存储（议题三）”奠定了坚实的架构基础。

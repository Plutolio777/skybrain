# SkyBrain 解决方案 V2 — 前置议题：竞品分析与核心壁垒设定

> 版本：V1.0
> 日期：2026-05-14
> 目标：分析市场现有解法，确立 SkyBrain 的设计底线与核心壁垒。

---

## 一、 市场现有解法分析（四大门派及代表产品）

在解决“AI 缺乏企业级项目上下文”这一痛点上，目前业界已经从早期的“把文档塞进 RAG”演进出了多种高维度的上下文引擎/数据库方案。我们将市场上现有的解决方案归纳为四大门派：

### 1. ContextOps（上下文运维）与包管理器平台
**核心解法**：侧重上下文的生命周期管理（创建->分发->维护），将“上下文”视为一种可被复用和分发的包。
*   **Tessl (闭源商业)**：定位为“Agent 技能的包管理器”，支持版本化管理 API 规范和架构约定，并分发给 Claude Code、Cursor 等。
*   **Packmind (闭源商业)**：侧重于从开发者的代码审查、架构决策中捕获团队特定的业务逻辑和规范，为 AI 提供人类知识维度的聚合。
*   **Ruler (闭源商业)**：企业级上下文指令集中管理平台。
*   **局限性**：偏向静态规则的“分发管道”，难以根据当前代码的编辑状态做到毫秒级的精准按需动态换页。

### 2. 持续学习与生产追踪引擎 (Continual Learning from Traces)
**核心解法**：通过记录开发者与 AI 的历史交互日志，提炼踩坑/填坑经验，形成记忆。
*   **Applied Compute Context Engine (商业)**：记录 Cursor/Claude Code 的所有日志（Traces），通过清洗去重构建 "Contextbase"。下次遇到类似问题时通过 MCP 注入。
*   **Mem0 (原 Embedchain) (开源 Apache-2.0, [GitHub](https://github.com/mem0ai/mem0))**：跨会话的个性化长期记忆层。自动提取实体、偏好和事实存入向量/图数据库。
*   **LangMem (开源 MIT, [GitHub](https://github.com/langchain-ai/langmem))**：LangChain 生态，后台不断将短期对话历史提炼为长期画像和核心规则。
*   **局限性**：严重依赖历史对话数据的累积（冷启动成本高），且对于明确的业务规则（如 Jira 需求、确定性架构规范）缺乏强管控输入。

### 3. 多智能体编译流水线与图谱架构 (Compiled View & Graph)
**核心解法**：后台存储复杂的图谱或状态机，在注入给 LLM 时“编译”成最小、最精准的工作上下文。
*   **Zep Graphiti (开源 Apache-2.0, [GitHub](https://github.com/getzep/graphiti))**：动态知识图谱。将非结构化代码和对话转化为节点和边（如函数调用关系），通过图查询返回具有拓扑关系的高质量上下文。
*   **Google ADK (实验性开源)**：将上下文视为“编译视图”。后台维持 Session 和 Artifact，运行时根据配置动态编译出当前 Agent 需要的最小 Context。
*   **Letta (原 MemGPT) (开源 Apache-2.0, [GitHub](https://github.com/letta-ai/letta))**：采用类似操作系统的虚拟内存管理，允许 Agent 自主将旧上下文“分页”写出，新上下文读入，对抗长对话遗忘。
*   **局限性**：多为底层开发框架，需要企业投入大量研发资源组装业务逻辑，非开箱即用的产品。

### 4. 显式引入与主动探索 (IDE 极客插件)
**核心解法**：不依赖庞大的后台知识库，依靠开发者手动指定或让 Agent 自行翻阅代码库。
*   **Continue.dev (开源 Apache-2.0, [GitHub](https://github.com/continuedev/continue))**：支持多 IDE。开发者通过 `@` 手动引入文件/文档，结合本地 AST 和 RAG 生成 prompt。
*   **Cline (开源 MIT, [GitHub](https://github.com/cline/cline))**：动态工具探索机制。不预加载，而是让 LLM 通过执行终端命令、查看特定行等工具“按需”获取上下文，保持极高专注度。
*   **Aider (开源 Apache-2.0, [GitHub](https://github.com/paul-gauthier/aider))**：采用 Repo Map（基于 tree-sitter 解析 AST 生成的高浓缩代码地图），以极少 Token 让 LLM 理解全局拓扑。
*   **局限性**：缺乏企业全局视野，没有跨人员的 RBAC 权限隔离、Jira 孤岛打通以及团队级的统一分发后台。

## 二、 SkyBrain 的核心壁垒与设计底线

基于对竞品局限性的分析，结合团队实际痛点，我们为 SkyBrain 确立了三条不可妥协的设计底线，这将构筑其核心护城河：

### 壁垒一：绝不脱离 IDE 主战场（沉浸式体验）
开发者极度厌恶上下文切换。SkyBrain 不论后端架构多么庞大，其最终的知识出口必须无缝融入开发者日常使用的 IDE 环境中。开发者无需跳出当前编辑窗口去网页端提问，系统应当能在他们键入代码或唤起 AI 时，自动准备好所需的弹药。

### 壁垒二：跨 AI 工具的“上下文协议层”（CSP 枢纽）
企业不会永远绑定单一 AI 工具。SkyBrain 必须扮演类似 LSP（Language Server Protocol）的角色，成为企业内的 **CSP（Context Server Protocol）枢纽**。它向下屏蔽各 IDE 和 AI 工具（Cursor, Trae, Copilot）的格式差异，向上提供统一的知识管理、权限控制和数据度量。

### 壁垒三：“冷热分层”与“多形态”的知识本体模型（类脑架构）
这是 SkyBrain 与传统静态 RAG 系统的本质区别。知识不应该被一股脑塞给大模型，也不应该仅仅是扁平的文本段落。SkyBrain 的大脑必须具备以下特征：

#### 1. 知识的冷热分层（L1 与 L2 缓存机制）
*   **L1 常驻上下文（潜意识底座）**：不管当前执行什么任务，AI 必须时刻具备的全局视野。包括：项目背景、宏观模块划分、全局基础编码规范、目录结构约定。这些知识必须常驻在 AI 的会话上下文中。
*   **L2 动态上下文（按需技能库）**：特定场景下才需要的专业知识。例如：如何连接特定的 ES 集群、如何使用内部封装的鉴权 SDK。这些知识仅在 AI 需要执行特定任务（如开发 API、编写查询）时，才被按需检索并换页加载到上下文中。

#### 2. 知识的多模态表现形态
虽然底层对 LLM 而言均表现为 Prompt，但在系统管理和调度层面，知识必须被明确分类并赋予不同的结构：
*   **Rules（规则）**：**约束型知识**。强调“不可违背的底线”（如命名规范、必须包含的日志 TraceID）。
*   **Skills（技能）**：**能力型知识**。强调“如何做某事”（如内部 SDK 的调用范例代码 + 参数说明）。
*   **Workflows（工作流）**：**流程型知识**。强调“标准操作 SOP”（如：新建一个完整 API 需要先定义 DTO，再写 Controller，最后补齐特定格式的单测）。

---

## 三、 总结

SkyBrain 的定位不是一个简单的“Markdown 托管后台”，而是一个**具备类脑分层记忆机制、支持多模态知识表达、且无缝桥接外部工作流与底层 IDE 的企业级 AI 研发上下文枢纽**。

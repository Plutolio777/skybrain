# SkyBrain 解决方案 V2 — 脑洞探索：L1/L2/L3 三层知识分级与双通道投递架构

> 状态：已修订（L1/L2 升级为 L1/L2/L3 三层）
> 阶段：V2 解决方案与产品雏形
> 核心问题：SkyBrain 的知识应该按什么维度分级投递给 AI？为什么不是简单地把所有知识一股脑塞进去，也不是单纯靠 MCP 按需查询？

---

## 一、 问题缘起：MCP 不靠谱 + L1 大杂烩的双重困境

在探讨 SkyBrain 如何将知识投喂给大模型时，遇到了两个层面的问题：

### 困境一：纯 MCP 的致命缺陷

> MCP 依赖大模型主动调用。对于"绝不能违背的基础规范"，大模型可能会因为"过度自信"或"注意力偏差"而拒绝查询。如果大模型认为"我知道怎么写支付接口"就跳过了查知识库，那 SkyBrain 就形同虚设。

| 方案 | 纯 MCP | 纯 .cursorrules |
|------|--------|-----------------|
| 覆盖面 | 全部知识可查 | 仅能容纳有限 Token |
| 可靠性 | 大模型可能不调用 | 强制注入，100% 送达 |
| 违背后果 | 偏门规范缺失 | Token 不足，长尾知识挤不进来 |

### 困境二：最初 L1 是个"大杂烩"

在早期设计中，L1 静态注入文件里混入了两类完全不同的东西：
- **元指令**（"写代码前必须通过 MCP 查询 SkyBrain 知识库"）—— 这是关于"如何使用 SkyBrain 本身"的知识
- **项目知识**（项目概述、技术栈选型、编码公约）—— 这是关于"当前项目"的知识

把它们混在一起有 3 个问题：
1. **元指令是跨项目通用的**，但被绑定在单个项目的 `.cursorrules` 里，换了项目要重写
2. **元指令和项目知识的更新节奏不同** —— 元指令很少变，项目知识随着团队规范演进频繁更新
3. **开发者分不清"哪些是 SkyBrain 强制我遵守的"和"哪些是团队自己定的"**

## 二、 核心设计理念：三层知识 + 双通道投递

### 三层知识定义

| 层级 | 知识类型 | 内容举例 | 投递通道 | 是否需同步到本地磁盘 |
|---|---|---|---|---|
| **L1** | SkyBrain 元知识 | "写代码前必须通过 MCP 查询 SkyBrain 知识库"、"遇到重构任务请先调用 SkyBrain 的 architecture skill"、"生成 API 接口前先查 L3 中是否有对应的 API 规范" | 静态文件（写磁盘） | ✅ 是 |
| **L2** | 项目级固化知识 | 项目概述、核心技术栈版本、全局编码公约、模块划分字典、强制安全规范 | 静态文件（写磁盘） | ✅ 是 |
| **L3** | 动态扩展知识 | API 定义、SDK 调用范例、长篇业务流（Workflows）、数据库表结构 | MCP 动态查询 | ❌ 否（实时查云端） |

### 双通道投递

| 通道 | 承载层级 | 机制 | 特点 |
|---|---|---|---|
| **通道一：磁盘写入** | L1 + L2 | Local Daemon 将 L1 元知识 + L2 固化知识合并渲染为目标 IDE 的静态规则文件（如 `.cursorrules`） | 强制注入 System Prompt，大模型无权拒绝 |
| **通道二：MCP 查询** | L3 | Local Daemon 作为 MCP Server，暴露 `search_skybrain_skills()` 等工具 | 按需拉取，不占上下文，由 L1 中的元指令驱动触发 |

### L1 和 L2 为什么都需要进磁盘？

L1（元知识）和 L2（项目知识）虽然内容性质不同，但它们有一个共同点：

> **必须 100% 送达，不能被大模型跳过。**

- L1 元知识如果不送达 → 大模型根本不会去查 L3 → 整个 SkyBrain 形同虚设
- L2 项目知识如果不送达 → 大模型缺乏项目背景 → 生成不符合规范的代码

只有 L3 适合走 MCP：它们庞大、专业、只有在特定场景下才需要，按需查询是最优解。

## 三、 最终方案：三层知识 + 双通道详解

### 3.1 通道一：L1 + L2 → 磁盘写入（强制注入）

**L1 元知识 —— 跨项目通用，SkyBrain 自带**

这是 SkyBrain 内置的一套"使用说明书"，告诉 AI 如何在任何项目中正确使用 SkyBrain：

```
## SkyBrain L1 Meta-Instructions (auto-generated, do not edit)

### CRITICAL: SkyBrain Knowledge Lookup Protocol

You MUST search SkyBrain before writing any code related to:
- API endpoints → call search_skybrain_skills("api")
- Database schemas → call search_skybrain_skills("database")
- Architecture decisions → call search_skybrain_skills("architecture")
- Business workflows → call search_skybrain_skills("workflow")
- SDK/third-party integration → call search_skybrain_skills("sdk")

If SkyBrain returns relevant knowledge, you MUST incorporate it into your response.
If SkyBrain returns no results, proceed with your best judgment.
```

**L2 固化知识 —— 项目专属，由团队 TL 管理**

这是项目团队的集体智慧沉淀，包含项目最基本的"常识"：

```
## SkyBrain L2 Project Knowledge (synced from SkyBrain Cloud)

### Project Overview
- 项目名称：OrderSystem
- 定位：B2B 订单管理系统
- 技术栈：Spring Boot 3.2 + MyBatis-Plus + MySQL 8.0 + Redis 7

### Team Conventions
- 所有 Controller 层代码统一放在 src/main/java/com/xxx/controller/
- Service 层禁止直接操作 HttpServletRequest
- 异常统一抛出 BusinessException(code, msg)
- 日志使用 @Slf4j，禁止 System.out.println
- 数据库操作必须使用逻辑删除（is_deleted 字段）

### Security Constraints
- 所有对外接口必须经过 JWT 鉴权
- 敏感字段（手机号、身份证号）必须加密存储
- SQL 参数必须使用 PreparedStatement，禁止拼接
```

### 3.2 通道二：L3 → MCP 动态查询（按需拉取）

L3 知识通过 MCP Server 暴露，AI 受 L1 元指令驱动后主动查询：

```
AI 写代码时的决策链：
  1. L1 元知识(磁盘)  → "写 API 前必须查 SkyBrain"
  2. L2 项目知识(磁盘) → "这是 OrderSystem, 用 Spring Boot"
  3. L3 MCP 查询     → AI 调用 get_api_spec("POST /order/payment")
  4. L3 返回          → 完整的支付接口签名、鉴权要求、示例代码
  5. AI 生成          → 完全符合规范的代码
```

### 3.3 L1 与 L2 存储位置的分离考量

| 维度 | L1 元知识 | L2 固化知识 |
|---|---|---|
| 来源 | SkyBrain 系统内置 | 云端团队大脑 |
| 变更频率 | 极低（SkyBrain 版本升级时） | 中等（团队规范演进时） |
| 跨项目通用 | ✅ 是 | ❌ 否 |
| 存储位置 | Local Daemon 安装目录 或 全局用户配置 | 云端拉取 + 写入项目规则文件 |

**初期简化方案**：L1 和 L2 合并写入同一个项目规则文件（如 `.cursorrules`），用分隔注释区分。以后版本可考虑 L1 走全局配置文件，L2 走项目文件。

## 四、 方案面临的核心挑战与应对策略

### 挑战 1：L1 + L2 的 Token 预算分配

**问题**：静态规则文件有长度预算。L1（约 500 tokens）+ L2（可能 2000-4000 tokens），加起来不能挤占太多用户对话上下文。

**应对**：
- L1 保持极简 —— 只写最关键的元指令，不带任何冗余
- L2 只放"确实不能遗漏"的固化知识 —— 能放 L3 的长篇细节（如完整的 API 文档）绝不放 L2
- Local Daemon 内置 Token 估算器，超限时给出裁剪建议

### 挑战 2：L1 更新后如何传播

**问题**：SkyBrain 版本升级后 L1 元指令变了，如何让所有开发者更新？

**应对**：
- L1 由 Local Daemon 自身管理，Daemon 升级时自动更新 L1 内容
- L1 更新不依赖云端同步，走 Daemon 版本升级通道

### 挑战 3：大模型"偷懒"跳过 MCP 调用

**问题**：即使 L1 写了"必须调用 MCP"，大模型仍然可能因为各种原因不调用。

**应对**：
- L1 元指令需要精心设计措辞，使用强约束语言（"You MUST search SkyBrain before writing any code related to..."）
- 可选的"二次确认"机制：无 MCP 调用的会话，SkyBrain 可以检测并提示开发者

### 挑战 4：IDE 差异适配

**问题**：不同 IDE 的静态规则文件路径不同。

**应对**：
- L1 + L2 同步器内置多 IDE 适配层（详见"多 IDE/多端适配策略"脑洞）

## 五、 关于"作用域"的维度澄清

知识分级与知识作用域是**两个正交的维度**：

| 层级 | 是否有作用域？ | 作用域范围 |
|---|---|---|
| **L1** 元知识 | ❌ 无 | 对整个 SkyBrain 系统有效，全局通用 |
| **L2** 固化知识 | ✅ 有 | Org → Project → Module → Branch（四级作用域，继承覆盖） |
| **L3** 动态知识 | ✅ 有 | 同 L2，按作用域精准投递 |

> L2 和 L3 的作用域分发机制详见"知识四级作用域与继承分发体系"脑洞文档。

## 七、 结论与后续行动

本次修订将知识从原来的 L1/L2 双轨升级为 L1/L2/L3 三层分级：

1. **L1 元知识**（磁盘写入）：SkyBrain 系统内置的"使用说明书"，跨项目通用，告诉 AI 如何正确使用 SkyBrain
2. **L2 固化知识**（磁盘写入）：项目级核心知识，由团队 TL 管理，确保 AI 拥有最基本的项目常识
3. **L3 动态知识**（MCP 查询）：长尾扩展知识，按需拉取，不占上下文

**关键洞察**：
- L1 + L2 共享"磁盘写入"通道，因为它们都需要**强制送达**
- L3 走独立的 MCP 通道，因为它是**按需扩展**的
- L1 是 L3 能正常工作的前提 —— 没有 L1 的元指令驱动，大模型根本不会想到去查 L3
- L1 无作用域（全局通用），L2 和 L3 有作用域（Org → Project → Module → Branch）

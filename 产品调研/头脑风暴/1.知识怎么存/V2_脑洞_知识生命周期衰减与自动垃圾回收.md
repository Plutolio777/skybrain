# SkyBrain 解决方案 V2 — 脑洞探索：知识生命周期衰减与自动垃圾回收（GC）

> 状态：讨论完成
> 阶段：V2 解决方案与产品雏形
> 核心问题：知识不是一劳永逸的。随着项目演进，曾经正确的规则会腐化。SkyBrain 如何感知知识过期？如何自动清理"知识垃圾"？

---

## 一、 问题缘起：知识腐化是知识库的"隐形癌症"

在前置议题中我们确立了知识的录入（议题二）与消费机制。但有一个致命问题被忽略了：

> 项目在演进，代码在重构，架构在升级。上个月 TL 审批通过的《支付模块开发规范》，这个月可能已经被新的微服务架构彻底推翻。如果 SkyBrain 继续把这条过期规则喂给 AI，AI 就会基于错误前提生成代码，产生大量毫无意义的返工。

**知识腐化的危害链**：过期知识 → AI 生成错误代码 → 开发者信任崩塌 → 不再使用 SkyBrain → 整个系统废弃。

因此，知识的生命周期管理（衰减 + 垃圾回收）不是锦上添花，而是**系统可持续运行的生存底线**。

---

## 二、 核心设计理念：手动标注为主力 + 自动衰减做兜底

经过深入讨论，我们确定了知识 GC 的组合策略：

1. **手动标注**（主力）：让知识消费者（开发者）主动评价知识质量，信号最干净
2. **自动衰减**（兜底）：对于无人评价、无人使用的"僵尸知识"，系统自动降权

两条路径互为补充：手动标注解决"信号准确度"问题，自动衰减解决"沉默沉没"问题。

---

## 三、 反馈感知方案探索：SkyBrain 如何知道"这条知识不行"？

### 3.1 三种探测探针（Probe A / B / C）

在脑暴初期，提出了三种知识过期判定机制：

**Probe A — 行为数据反馈（已采用）**
- 原理：基于"开发者与 AI 的实际交互反馈"——当 AI 基于某条知识生成代码后，开发者手动修改了大量内容，说明知识可能有问题
- 核心挑战：代码修改发生在 IDE 中，SkyBrain 如何感知？

**Probe B — AST 结构偏移物理探测（备选，后期实现）**
- 原理：系统定期扫描代码库。如果 Rule_Y 说"所有 Controller 必须继承 BaseAbstractController"，但最新提交的新 Controller 已不再继承该类 → 标记为 `[Suspicious]`
- 适用场景：编码规范、架构约束类知识的自动校验

**Probe C — 外部数据源级联失效（备选，后期实现）**
- 原理：知识携带来源标签 `<Source: Jira-ISSUE-1234>`。当该 Jira 任务被关闭 / 作废时，所有派生知识级联标记为失效
- 适用场景：从 Jira / Confluence / 设计文档中提取的知识

### 3.2 Probe A 的核心攻坚：反馈感知路径

Probe A 的最大工程难点在于——开发者对 AI 代码的"负反馈"发生在 IDE 内部，SkyBrain 作为外部守护进程，如何感知？

经过对四条潜在路径的逐一分析：

| 路径 | 机制 | 信号强度 | 评价 |
|------|------|---------|------|
| Git Diff 事后对比 | AI 生成时做快照 → commit 后 diff 对比修改率 | 强 | 滞后，无法感知未 commit 的修改 |
| MCP 反馈通道 | AI 工具通过 MCP 回传 accept/modify/reject | 最强 | 依赖 AI 工具生态适配，短期不可控 |
| 文件系统监听 | watcher 监听 AI 涉及文件的变化 | 弱 | 噪声太大，无法区分"改 bug"和"改 AI 的错" |
| Git Blame / Author 分析 | 检测同一开发者对 AI 生成代码的快速二次提交 | 中 | 零侵入，但只能感知"有修改"，不知"为什么改" |

**结论**：纯自动化的行为感知路径都存在精度或时效性问题，最终确立以**手动反馈**为核心感知手段。

## 四、 最终方案：双通道反馈架构

最终确定的反馈采集方案为**双通道并行**：

### 通道一：MCP 会话内即时反馈（VibeCoding 期间）

- **位置**：MCP Server 接口层
- **触发时机**：AI 会话进行中或刚结束时
- **交互形态**：
  - 开发者在 AI 对话中可以自然表达："刚才那条重试规范不对，已经过时了" → MCP 的 LLM 解析语义，自动标注对应知识
  - 或通过 MCP 暴露的轻量级工具：`rate_knowledge(id, 👍/👎)` 
- **优势**：在开发流中完成，不打断心流
- **采样策略**：不每次必弹，每消费 10 次知识才提示一次评价机会，避免反馈疲劳

### 通道二：Local Daemon 面板手动标注（事后回顾）

- **位置**：SkyBrain 本地守护进程面板
- **触发时机**：开发者主动打开 Local Daemon 管理界面时
- **交互形态**：
  - 按"知识条目"级别，开发者可以看到每一段拉取过的知识
  - 每条知识旁边提供 👍 / 👎 一键标注
  - 可选填写一句话说明"为什么不对"
- **优势**：针对性最强，可以逐条精准评价
- **粒度选择**：采用**知识条目级别**标注（而非文档级别），因为一篇文档可能 90% 正确但某一小段过时

## 五、 方案面临的核心挑战与应对策略

### 挑战 1：反馈疲劳

**问题**：如果每次 AI 会话结束都弹评分，第三天就没人填了。

**应对**：
- 不做"每次必评"，采用**抽样触发**：每消费 N 次知识（如 10 次）才触发一次评价邀请
- 交互极致轻量：👍 / 👎 一键反馈，不做多维度量表
- 通道一的"对话式反馈"更加自然，可替代结构化表单

### 挑战 2："沉默的差评"

**问题**：最危险的负反馈是——开发者发现知识过时，心里骂了一句，然后默默自己改代码，什么反馈都不给。

**应对**：
- 手动标注只捕获"主动反馈"，无法解决这个问题
- 必须依赖**自动衰减兜底机制**（见第六章）
- 长期可结合 Probe B（AST 偏移检测）作为补充信号

### 挑战 3：标注粒度

**问题**：一次 AI 对话拉取了 5 条 Rules + 2 个 Skills + 1 篇摘要。标注太粗（文档级）会误伤，太细（逐句级）会累死人。

**决策**：采用**知识条目级别**标注
- 每条 Rule / Skill / 摘要作为一个独立标注单元
- 不做更细粒度的拆分，平衡准确度与操作成本

### 挑战 4：冷启动与样本偏差

**问题**：
- 新知识上来就是 0 评价 → 不能让它一出生就被当"可疑"
- 只有 2 人评过，其中 1 人因个人理解错误给了差评 → 被冤枉

**应对**：
- 新知识默认给予**中等偏上的初始信任分**（"无罪推定"）
- 信任分采用**加权贝叶斯平均**，小样本时向默认信任分回缩（类似 IMDB 评分算法）
- 只有样本量达到阈值后，用户评分才起决定性作用

## 六、 双重引擎：手动标注 + 自动衰减的数据模型

### 6.1 知识健康度评分（Health Score）

每条知识维护一个健康度评分（0-100），由两个因子共同决定：

```
HealthScore = UserRatingScore × Weight_user + TimeDecayScore × Weight_auto
```

| 因子 | 来源 | 权重 |
|------|------|------|
| UserRatingScore | 双通道手动反馈（👍/👎） | 高（有足够样本时） |
| TimeDecayScore | 自然时间衰减 + 使用频率衰减 | 低（兜底作用） |

### 6.2 自动衰减规则

- **时间衰减**：知识超过 N 天（如 90 天）未被任何人检索/使用 → HealthScore 线性下降
- **低频衰减**：知识被检索但从未触发过 👍 → 缓慢降权（类似 LRU Cache 的冷数据驱逐）
- **触发阈值**：HealthScore 低于阈值 → 自动从 Global Brain 降级到 `[Outdated]` 待审区，通知原始作者

### 6.3 衰减后的处理策略

被判定为过期的知识，采用**逻辑删除而非物理删除**：

- **对 AI 隐藏**：过期知识不再出现在 MCP 检索结果中，AI 不可见
- **保留归档**：知识本体保留在系统归档中，TL 可以在 Web 端查看"已失效知识列表"
- **可复活**：如果架构回滚或规范恢复，TL 可以一键复活过期知识
- **理由**：物理删除不可逆，且历史知识对追溯架构演进有价值。逻辑删除保证"AI 不会被污染"的同时不丢失知识资产。

## 八、 结论与后续行动

本次脑洞探索明确了 SkyBrain 知识 GC 的核心策略：

1. **反馈采集**：双通道并行 — MCP 会话内自然语言反馈 + Local Daemon 面板逐条标注
2. **评分引擎**：手动标注（主力）与自动衰减（兜底）相结合的 HealthScore 模型
3. **处理策略**：逻辑删除（对 AI 隐藏，保留归档），结合三级生命周期中已有的"降级退回提案"流程
4. **Probe B / C**：作为后期增强方向，V1 阶段不强制实现

该机制与议题二的"知识三级生命周期"形成完整闭环：知识有"生"（录入 → 审核 → 生效），也必须有"死"（衰减 → 降级 → 逻辑删除），才能保证 SkyBrain 大脑的长期健康。

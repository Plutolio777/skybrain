# SkyBrain 解决方案 V2 — 脑洞探索：知识内容形态与生效范围分类模型

> 状态：讨论中
> 阶段：V2 解决方案与产品雏形
> 核心问题：SkyBrain 作为 AI 编程上下文数据库，需要存储哪些类型的知识？这些知识应该如何分类？从内容形态应该如何推导出分级加载体系？

---

## 一、问题缘起：产品设计的起点应该是"存什么"

在设计一个知识库系统时，最自然的起点是回答：

> **"这个知识库里到底存什么？"**

只有先搞清楚"知识长什么样"，才能推导出"什么时候喂给 AI"、"通过什么通道喂"、"怎么组织检索"。如果反过来用投递通道去套内容分类，就会陷入"因为它是 L3，所以它是 L3"的循环论证。

因此，本次脑洞采用以下推导链路：

```
内容形态（知识长什么样）
     ↓ 推导
生效范围（什么时候需要它）
     ↓ 推导
分级加载体系（L1/L2/L3 —— 通过什么通道喂给 AI）
```

---

## 二、第一步：知识的内容形态（四大类九种）

定义"这条知识承载了什么内容"：

```
SkyBrain 知识（按内容形态）
│
├── 📋 操作型知识（How to）        → 告诉 AI "怎么做"
│   ├── Skill             特定场景下的代码级操作范例
│   ├── Workflow          跨模块/跨步骤的完整业务流程
│   └── Reference         外部依赖或工具库的使用说明
│
├── 📐 契约型知识（What is）       → 告诉 AI "是什么"
│   ├── API Spec          接口/方法签名与契约定义
│   └── Schema            数据库表结构/数据模型定义
│
├── 📖 认知型知识（Context）       → 告诉 AI "在哪、是什么项目"
│   ├── Architecture      项目/模块的整体架构概述
│   └── Glossary          业务术语和领域概念的定义
│
└── ⚖️ 约束型知识（Must follow）  → 告诉 AI "绝不能做什么"
    ├── Rule              硬性安全/技术约束（不可违背）
    └── Convention        团队编码风格/命名约定
```

### 九类知识详解

| 内容形态 | 知识类型 | 定义 | 示例 |
|----------|----------|------|------|
| 📋 操作型 | Skill | 特定场景下的代码级操作范例 | "如何在 OrderService 中实现支付回调鉴权" |
| 📋 操作型 | Workflow | 跨模块/跨步骤的完整业务流程 | "订单退款：用户发起 → 审核 → 调用支付网关 → 回滚库存" |
| 📋 操作型 | Reference | 外部依赖的技术栈使用说明 | "本项目使用 dayjs 替代 moment.js，导入路径 @/utils/dayjs" |
| 📐 契约型 | API Spec | 接口/方法的完整签名和契约定义 | "POST /order/payment，参数 OrderPaymentReq，返回 PaymentResult" |
| 📐 契约型 | Schema | 数据库表结构/数据模型定义 | "orders 表：id, user_id, amount, status(enum), is_deleted" |
| 📖 认知型 | Architecture | 项目/模块的整体架构概述 | "OrderSystem 采用 DDD 分层：Interface → Application → Domain → Infrastructure" |
| 📖 认知型 | Glossary | 业务术语和领域概念的精确定义 | "出库单（OutboundOrder）= 客户确认后生成的发货指令" |
| ⚖️ 约束型 | Rule | 必须遵守的硬性安全/技术约束 | "所有 SQL 参数必须用 PreparedStatement；对外接口必须 JWT 鉴权" |
| ⚖️ 约束型 | Convention | 团队编码风格和命名约定 | "Controller 类统一放在 src/.../controller/，类名以 Controller 结尾" |

## 三、第二步：从内容形态推导生效范围

确定了知识内容后，自然引出一个问题：**"这种知识在什么场景下需要被喂给 AI？"**

### 3.1 推导逻辑

我们逐一分析四大内容形态的使用场景：

**约束型知识（Rule / Convention）**

- Rule 描述的是"绝对不能违背的事"。
- Convention 描述的是"团队默认应该做的事"。
- → 这两类知识**不管 AI 在写什么代码都必须知道**。如果 AI 不知道"禁止 SQL 拼接"，它可能随时写出不安全的代码。
- → 结论：**全局常驻** —— 始终生效，没有例外。

**认知型知识（Architecture / Glossary）**

- Architecture 描述的是"项目长什么样"，是理解一切代码的前置常识。
- 如果 AI 不知道项目的分层架构，它连文件该放哪都搞不清。
- Glossary 描述的是业务术语的含义，但具体术语只在相关模块中出现。
- → Architecture 必须**全局常驻**；Glossary 可以**按模块绑定**。
- → 结论：**全局常驻为主 + 模块绑定为辅**。

**契约型知识（API Spec / Schema）**

- API Spec 和 Schema 都与特定模块强相关。编辑订单模块时才需要订单的 API 和表结构。
- 写前端代码时读到后端表结构，纯属浪费 Token。
- → 结论：**模块绑定** —— 编辑哪个模块就喂哪个模块的契约。

**操作型知识（Skill / Workflow / Reference）**

- Skill 和 Workflow 是"怎么做某件事"的范例，只有 AI 准备做那件事时才需要。
- Reference 虽然是操作型，但它描述的是"这个项目用什么依赖"，跟具体文件路径相关。
- → Skill/Workflow 是**意图触发**（AI 识别到"写支付逻辑"时才查），Reference 是**模块绑定**。
- → 结论：**意图触发为主 + 模块绑定为辅**。

### 3.2 生效范围三类定义

```
生效范围
├── 🌐 全局常驻    → 一旦激活 SkyBrain，这条知识就必须对 AI 可见
├── 📦 模块绑定    → 只有当开发者编辑特定模块的文件时，才注入
└── 🔗 意图触发    → 只有当 AI 识别到特定任务意图时，才主动检索
```

### 3.3 全量映射表

| 内容形态 | 知识类型 | 默认生效范围 | 推导理由 |
|----------|----------|-------------|----------|
| ⚖️ 约束型 | Rule | 🌐 全局常驻 | 安全红线，任何场景都不能遗漏 |
| ⚖️ 约束型 | Convention | 🌐 全局常驻 | 编码风格，所有代码都应遵守 |
| 📖 认知型 | Architecture | 🌐 全局常驻 | 项目架构是理解一切代码的前置常识 |
| 📖 认知型 | Glossary | 📦 模块绑定 | 业务术语只在其所属模块上下文中出现 |
| 📐 契约型 | API Spec | 📦 模块绑定 | 接口定义与模块强绑定，无需跨模块注入 |
| 📐 契约型 | Schema | 📦 模块绑定 | 表结构只在涉及该数据表的编辑场景下需要 |
| 📋 操作型 | Reference | 📦 模块绑定 | 依赖用法与具体引用文件路径相关 |
| 📋 操作型 | Skill | 🔗 意图触发 | 只有 AI 准备写那类功能时才需要看范例 |
| 📋 操作型 | Workflow | 🔗 意图触发 | 只有 AI 被要求实现完整业务链路时才需要 |

### 3.4 生效范围的规律总结

```
约束型 + 认知型  ──→ 「全局常驻」为主
  ↳ 它们是 AI 的"骨骼和常识"，缺失任何一条都会导致系统级错误
  ↳ 例外：Glossary 虽然是认知型，但在 L3 中按模块绑定更高效

契约型        ──→ 「模块绑定」为主
  ↳ 它们是特定模块的"接口契约"，跨模块注入是噪音

操作型        ──→ 「意图触发」为主
  ↳ 它们是"按需查阅的工具书"，不需要时不应占用上下文
  ↳ 例外：Reference 虽然是操作型，但更贴近模块绑定
```

## 四、第三步：从生效范围推导 L1/L2/L3 分级加载体系

确立了"什么知识在什么场景需要"之后，下一个问题就是：**"怎么让它在那个场景下正好出现？"**

### 4.1 推导逻辑

**🌐 全局常驻型知识 —— 必须 100% 送达，大模型不能跳过**

这类知识（Rule、Convention、Architecture）如果大模型没读到，整个 SkyBrain 就形同虚设。而唯一能保证 100% 送达的方式，就是**写入磁盘静态文件**，利用 IDE 自带的 System Prompt 注入机制强制加载。

- → 必须走**磁盘写入通道**

**📦 模块绑定型 + 🔗 意图触发型 —— 按需出现，太多了会撑爆上下文**

这类知识（API Spec、Schema、Skill、Workflow 等）数量庞大、专业性强，如果全量写入静态文件，Token 预算瞬间耗尽。它们需要的是"在正确的时间精确出现"——也就是**动态按需检索**。

- → 必须走**MCP 动态查询通道**

### 4.2 三层分级体系的诞生

从以上推导，自然产生了两种投递通道（磁盘写入 / MCP 查询）。但进一步细分，磁盘写入通道内的知识还有两种不同性质：

- **SkyBrain 自身的"使用说明书"**（告诉 AI 该怎么用 SkyBrain）—— 这是跨项目通用的元知识
- **当前项目的"基本常识"**（Rule、Convention、Architecture）—— 这是项目专属的固化知识

两者虽然都走磁盘写入，但来源、更新频率、跨项目通用性完全不同，必须分离。

由此，完整的 **L1/L2/L3 三层分级加载体系** 便从内容形态自然推导而出：

```
            第三步推导结果：L1 / L2 / L3 分级加载体系
    ┌─────────────────────────────────────────────────────┐
    │                                                     │
    │  L1 元知识                                  L2 固化知识
    │  ┌─────────────────────┐  ┌───────────────────────┐│
    │  │  SkyBrain 自身       │  │  从内容形态中推导     ││
    │  │  使用说明书           │  │  ↓                   ││
    │  │  (系统内置)           │  │  全局常驻型知识       ││
    │  │                     │  │  · Rule               ││
    │  │  例："写 API 前      │  │  · Convention         ││
    │  │  必须先查 SkyBrain"  │  │  · Architecture       ││
    │  └─────────────────────┘  └───────────────────────┘│
    │              │                       │              │
    │              └───────┬───────────────┘              │
    │                      ↓                              │
    │              通道一：磁盘写入                         │
    │          (渲染为 .cursorrules 等静态文件)             │
    │          强制注入 System Prompt, 大模型无权拒绝       │
    │                                                     │
    └─────────────────────────────────────────────────────┘
                                +
    ┌─────────────────────────────────────────────────────┐
    │                                                     │
    │  L3 动态扩展知识                                     │
    │  ┌───────────────────────────────────────────────┐  │
    │  │  从内容形态中推导                               │  │
    │  │  ↓                                            │  │
    │  │  模块绑定型知识          意图触发型知识          │  │
    │  │  · API Spec             · Skill               │  │
    │  │  · Schema               · Workflow            │  │
    │  │  · Glossary                                      │  │
    │  │  · Reference                                     │  │
    │  └───────────────────────────────────────────────┘  │
    │                      ↓                               │
    │              通道二：MCP 动态查询                      │
    │    (Local Daemon 作为 MCP Server, AI 主动拉取)        │
    │             按需加载, 不占上下文                       │
    │                                                     │
    └─────────────────────────────────────────────────────┘
```

### 4.3 完整推导链路总结

```
内容形态                         生效范围              分级加载
───────────────────────────────────────────────────────────────
Rule           (约束型)        ──→ 🌐 全局常驻 ──→  L2 磁盘写入
Convention     (约束型)        ──→ 🌐 全局常驻 ──→  L2 磁盘写入
Architecture   (认知型)        ──→ 🌐 全局常驻 ──→  L2 磁盘写入
Glossary       (认知型)        ──→ 📦 模块绑定 ──→  L3 MCP 查询
API Spec       (契约型)        ──→ 📦 模块绑定 ──→  L3 MCP 查询
Schema         (契约型)        ──→ 📦 模块绑定 ──→  L3 MCP 查询
Reference      (操作型)        ──→ 📦 模块绑定 ──→  L3 MCP 查询
Skill          (操作型)        ──→ 🔗 意图触发 ──→  L3 MCP 查询
Workflow       (操作型)        ──→ 🔗 意图触发 ──→  L3 MCP 查询

+ L1 元知识（系统内置，独立于内容形态）
  ──→ 跨项目通用 ──→ L1 磁盘写入
```

### 4.4 L1 元知识的特殊性

L1 元知识（如"写 API 前必须先查 SkyBrain 的 L3 知识库"）**不是从内容形态推导出来的**。

它是 SkyBrain 系统自身的组成部分 —— 它的作用是告诉 AI **"如何正确使用 SkyBrain"**。它与 L2 虽然同走磁盘写入通道，但有几个本质区别：

| 维度 | L1 元知识 | L2 固化知识 |
|------|----------|------------|
| 来源 | SkyBrain 系统内置 | 团队录入 + 冷启动扫描 |
| 变更频率 | 极低（跟随 Daemon 版本升级） | 中等（团队规范演进） |
| 跨项目通用 | ✅ 是 | ❌ 否 |
| 设计目的 | 驱动 AI 去查询 L3 | 告诉 AI 项目的基本常识 |

## 五、方案面临的核心挑战与应对策略

### 挑战 1：知识的生效范围并非完全固定

**问题**：某条知识可能同时适配多个生效范围。比如某个支付鉴权 Skill 既属于操作型知识（意图触发），又跟 order 模块强绑定（模块触发）。一个安全 Rule 虽然是全局常驻，但如果某个子模块有更严格的特化规则，需要局部覆盖。

**应对**：采用"默认值 + 手动覆盖"策略 ——
- 每种知识类型在创建时自动赋予默认生效范围
- TL 或创建者可以在知识属性中手动覆盖
- 知识点可以同时打上多个生效标签（如既是"意图触发"也是"模块绑定"）

### 挑战 2：模块绑定的粒度问题

**问题**：多 Module 的代码库目录结构千差万别，"模块"的边界如何定义？

**应对**：
- 支持通过项目级 `skybrain.yaml` 显式声明模块路径映射
- 若无声明，默认按顶级目录作为隐式模块划分
- 知识创建时可以由 AI 自动推断关联的文件 glob pattern

### 挑战 3：意图触发的匹配精度

**问题**：AI 说"写一个支付接口"，SkyBrain 需要识别这是"支付"意图并拉取相关 Skill。但如何区分"支付"和"退款"？误匹配会浪费 Token 或漏掉关键知识。

**应对**：
- 知识入库时生成语义向量，与 AI 当前任务描述做相似度匹配
- 每条知识维护一个"触发短语"列表，提升匹配精度
- 支持多级意图标签体系（一级：支付 → 二级：鉴权/回调/退款）

## 六、与四级作用域的关系

生效范围（全局/模块/意图）定义了知识"什么时候喂给 AI"，作用域（Org/Project/Module/Branch）定义了知识"归属于哪个组织单元"。两者是正交维度：

| | 🌐 全局常驻 | 📦 模块绑定 | 🔗 意图触发 |
|------|----------|----------|----------|
| **Org 级** | 公司级编码规约 | — | — |
| **Project 级** | 项目架构概述 | 项目通用 API | 跨模块 Workflow |
| **Module 级** | — | 模块专属 API Spec | 模块专属 Skill |
| **Branch 级** | — | 分支临时 Schema | 分支过渡方案 Skill |

## 七、待进一步探讨的问题

1. **Schema 细节**：每种知识类型的内部结构（字段）长什么样？（如 Skill 是否需要「前置条件」「适用场景」「示例代码」等固定字段？）
2. **分类的扩展性**：未来出现新的知识类型（如「测试策略」「部署配置」），如何嵌入现有的四大内容形态？
3. **多标签生效范围的冲突解决**：如果某知识同时标记了"模块绑定"和"意图触发"，以哪个为准？还是双重触发？

## 八、结论

本次脑洞确立了 SkyBrain 知识从内容到投递的**完整推导链路**：

> **第一步**：知识按内容形态分为四大类九种 —— 操作型（Skill/Workflow/Reference）、契约型（API Spec/Schema）、认知型（Architecture/Glossary）、约束型（Rule/Convention）。
>
> **第二步**：从内容形态推导出生效范围 —— 约束型和认知型天然要求全局常驻，契约型天然匹配模块绑定，操作型天然适合意图触发。
>
> **第三步**：从生效范围推导出 L1/L2/L3 分级加载体系 —— 全局常驻的走磁盘写入（L1/L2），确保 100% 送达；模块绑定和意图触发的走 MCP 动态查询（L3），实现按需精准投递，不浪费 Token。L1 元知识作为 SkyBrain 自身的"使用说明书"独立于内容形态之外。

这一推导链路确保了 SkyBrain 的架构设计**从产品逻辑出发，而非技术方案出发**：先回答"存什么"，再回答"什么时候出现"，最后回答"怎么让它出现"。

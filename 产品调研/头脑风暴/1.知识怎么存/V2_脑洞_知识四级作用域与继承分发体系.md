# SkyBrain 解决方案 V2 — 脑洞探索：知识四级作用域与继承分发体系

> 状态：讨论完成（已确认设计）
> 阶段：V2 解决方案与产品雏形
> 核心问题：知识应当如何与代码关联？单一的分支强绑定是否足够？如何保证不同粒度的知识（全局公约 vs 特定模块逻辑）能够精准地投递给 AI，既不污染上下文，也不遗漏关键信息？

---

## 一、 问题缘起：分支强绑定的局限性与"知识一刀切"困境

在早期的构想中，为了解决知识随代码演进的问题，我们提出了"知识与 Git 分支强绑定"的思路（即所有知识打上 `<ProjectID, Branch>` 标签）。但在深入推演后，发现了致命的逻辑漏洞：

1. **固有知识的越级属性**：像"项目概述"、"基础框架选型"、"全局编码规范"这类知识，是不随分支迭代而发生转移的。如果将它们绑定到 `feature-A` 分支，当开发者切换到 `feature-B` 时，这些基础知识就会丢失，导致 AI 重新出现"幻觉"。
2. **模块边界的上下文爆炸**：如果所有的项目级知识都在整个项目中生效，那么修改前端组件时，AI 也会被迫读取后端的数据库设计规范，白白消耗 Token 且容易引发混乱。
3. **版本演进与作用域的混淆**：知识的**适用范围**（它在哪里生效）与知识的**版本历史**（它的 V1、V2 演变）是两个正交（独立）的维度，不能简单地用 Git 分支来同时指代这两者。

---

## 二、 核心设计理念：多维正交与四级作用域树

基于上述发现，SkyBrain 决定将**知识的作用域（Scope）**与**版本演进（Version）**解耦，并针对作用域引入一套**基于层级继承的四级树状模型**。

### 理念 1：作用域与版本的正交性
- **作用域（Scope）**：决定这条知识在**空间上**影响哪些文件、哪些开发者（Org -> Project -> Module -> Branch）。
- **版本（Version）**：决定这条知识在**时间上**的演变历史。版本号的生成**不与 Git Commit 强行绑定**，而是由 TL 审批通过后生成正式的业务版本号（如规则 V1、规则 V2）。

### 理念 2：继承与覆盖原则
下层作用域（如 Module）会自动继承上层作用域（如 Project）的知识。当上下层存在冲突时，下层知识**优先覆盖**上层知识（就近原则）。

---

## 三、 最终方案：四级作用域分发架构

每条在 SkyBrain 中央大脑中的知识，都必须明确其所属的 `Scope` 级别，并包含相应的元数据：

### 1. 作用域层级定义

| 级别 | Scope | 说明 | 适用场景 | 路由投递条件 |
| --- | --- | --- | --- | --- |
| **L1** | `Org` (公司级) | 全公司/跨项目通用的底层规范 | 公司的 Java 编码规约、通用的鉴权设计标准 | 只要属于该组织的开发者，自动下发 |
| **L2** | `Project` (项目级) | 单个项目所有分支、所有模块通用的核心知识 | 项目概述、整体架构图、核心技术栈版本 | Local Daemon 检测到在当前工程目录下，自动下发 |
| **L3** | `Module` (模块级) | 项目中特定模块或子包专属的规则 | 前端 UI 组件库的使用规范、订单模块的特殊状态机逻辑 | Local Daemon 检测到开发者**当前正在编辑的文件路径**属于该模块，按需下发 |
| **L4** | `Branch` (分支级) | 仅针对某个特性分支/修复分支生效的临时知识 | 某个重构分支中的临时过渡方案、正在开发的未合并新特性说明 | Local Daemon 检测到当前的 Git Branch 匹配时，按需下发 |

### 2. Local Daemon 的情境感知（Context Awareness）

为了实现上述精准下发，部署在开发者本机的 Local Daemon 需要具备更强的情境感知能力：

1. **路径嗅探（Path Sniffing）**：Daemon 会实时监听开发者在 IDE 中活跃（Active）的文件路径。
   - *例子*：如果当前编辑的是 `src/modules/order/Payment.ts`，Daemon 会向云端请求 `Module = order` 及其父级（Project、Org）的知识。
2. **分支嗅探（Branch Sniffing）**：Daemon 监听 `.git/HEAD` 的变化，获取当前工作分支。
3. **上下文动态合成（Context Composition）**：Daemon 拉取到 `Org + Project + Module + Branch` 的知识集合后，根据**就近覆盖原则**解决冲突，最终合并为一个 `.cursorrules` 文件或通过 MCP 返回给 IDE。

### 3. 数据结构演进

知识实体的 Metadata 结构更新为：
```json
{
  "knowledge_id": "k_10023",
  "content": "订单模块支付回调必须验证签名...",
  "scope_level": "module",
  "scope_target": {
    "org_id": "org_778",
    "project_id": "proj_992",
    "module_path": "src/modules/order/*",
    "branch_name": null
  },
  "version": "v2.1", // 由 TL 审批后独立演进的版本号
  "health_score": 95
}
```

## 四、 挑战与应对

### 1. 模块边界如何界定？
- **挑战**：代码库的目录结构千差万别，如何准确定义一个 `Module` 的边界？
- **应对**：支持在项目级设置中通过 `skybrain.yaml` 显式声明模块路径映射；若无声明，则默认采用顶层目录（如 `src/components`, `src/services`）作为隐式模块划分。也可以在知识录入时，由 AI 根据开发者的意图自动推断关联的文件 glob pattern。

### 2. 跨模块调用的知识投递
- **挑战**：当在订单模块调用用户模块的接口时，需要知道用户模块的知识吗？
- **应对**：引入"引用链嗅探"。当 AST 分析到当前文件 Import 了其他模块的类时，Local Daemon 可以将被引用模块的知识作为"弱上下文"一并拉取（分配较低的优先级，防止 Token 超载）。

---

## 五、 结论与后续行动

通过将"分支绑定"升级为"四级作用域树（Org -> Project -> Module -> Branch）"，我们彻底解决了项目通用知识与局部特化知识的冲突问题，做到了知识的**按需加载、精准滴灌**。

**后续需进一步探讨的脑洞：**
- **知识的版本演进与变更历史**：既然版本（Version）已经与分支（Branch）解耦，那么当一条 Project 级的知识发生变更时，它的 V1 到 V2 是如何演进的？TL 审批动作如何触发版本升级？开发者能否查阅知识的历史 Diff？

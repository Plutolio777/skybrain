# SkyBrain 解决方案 V2 — 脑洞探索：从内容形态推导 L1/L2/L3 分级加载体系

> 状态：讨论中
> 阶段：V2 解决方案与产品雏形
> 核心问题：确定了知识的内容形态之后，如何从每种形态的特征出发，推导出"知识应该在什么时候、通过什么方式喂给 AI"？

---

## 一、推导起点

前置脑洞已确立了知识的四大内容形态分类：

```
操作型（Skill / Workflow / Reference）    → 告诉 AI "怎么做"
契约型（API Spec / Schema）              → 告诉 AI "是什么"
认知型（Architecture / Glossary）        → 告诉 AI "在哪、是什么项目"
约束型（Rule / Convention）              → 告诉 AI "绝不能做什么"
```

每种形态有截然不同的特征（体量、数量、变更频率、对 AI 的必要性）。这些特征是推导"投递方式"的天然依据。

---

## 二、第一步：从内容形态推导生效范围

确定知识内容后，自然引出一个问题：**"这种知识在什么场景下需要被喂给 AI？"**

### 2.1 逐一推导

**约束型知识（Rule / Convention）**

- Rule 描述的是"绝对不能违背的事"。
- Convention 描述的是"团队默认应该做的事"。
- → 不管 AI 在写什么代码，只要它不知道"禁止 SQL 拼接"，就可能随时写出不安全的代码。
- → 结论：**全局常驻**，始终生效，没有例外。

**认知型知识（Architecture / Glossary）**

- Architecture 描述项目结构和分层，是理解一切代码的前置常识。AI 不知道项目有 DDD 分层，文件都不会放对位置。
- Glossary 描述业务术语，具体术语只在相关模块上下文中出现。写支付模块不需要知道仓储模块的术语定义。
- → Architecture 必须**全局常驻**；Glossary 可以**按模块绑定**。

**契约型知识（API Spec / Schema）**

- 接口和表结构都与特定模块强绑定。编辑订单模块时只需要订单的 API 和表结构。
- 写前端代码时读到后端表结构，纯属浪费 Token。
- → 结论：**模块绑定**，编辑哪个模块就喂哪个模块的契约。

**操作型知识（Skill / Workflow / Reference）**

- Skill 和 Workflow 是"怎么做某件事"的范例，只有 AI 准备做那件事时才需要。
- Reference 描述的是"这个项目用什么依赖"，跟具体引用了该依赖的文件路径相关。
- → Skill/Workflow 是**意图触发**（AI 识别到任务意图时才查），Reference 是**模块绑定**。

### 2.2 生效范围三类定义

```
生效范围
├── 🌐 全局常驻    → 一旦激活 SkyBrain，这条知识就必须始终对 AI 可见
├── 📦 模块绑定    → 只有当开发者编辑特定模块的文件时，才注入
└── 🔗 意图触发    → 只有当 AI 识别到特定任务意图时，才主动检索
```

### 2.3 全量映射表

| 内容形态 | 知识类型 | 默认生效范围 | 推导理由 |
|----------|----------|-------------|----------|
| ⚖️ 约束型 | Rule | 🌐 全局常驻 | 安全红线，任何场景都不能遗漏 |
| ⚖️ 约束型 | Convention | 🌐 全局常驻 | 编码风格，所有代码都应遵守 |
| 📖 认知型 | Architecture | 🌐 全局常驻 | 项目架构是理解一切代码的前置常识 |
| 📖 认知型 | Glossary | 📦 模块绑定 | 业务术语只在其所属模块上下文中出现 |
| 📐 契约型 | API Spec | 📦 模块绑定 | 接口定义与模块强绑定，无需跨模块注入 |
| 📐 契约型 | Schema | 📦 模块绑定 | 表结构只在涉及该数据表的编辑场景下需要 |
| 📋 操作型 | Reference | 📦 模块绑定 | 依赖用法与具体引用文件路径相关 |
| 📋 操作型 | Skill | 🔗 意图触发 | 只有 AI 准备写那类功能时才需要看范例 |
| 📋 操作型 | Workflow | 🔗 意图触发 | 只有 AI 被要求实现完整业务链路时才需要 |

### 2.4 规律总结

```
约束型 + 认知型  ──→ 「全局常驻」为主
  ↳ 它们是 AI 的"骨骼和常识"，缺失任何一条都会导致系统级错误
  ↳ 例外：Glossary 虽然是认知型，但按模块绑定更高效

契约型        ──→ 「模块绑定」为主
  ↳ 它们是特定模块的"接口契约"，跨模块注入是噪音

操作型        ──→ 「意图触发」为主
  ↳ 它们是"按需查阅的工具书"，不需要时不应占用上下文
  ↳ 例外：Reference 虽然是操作型，但更贴近模块绑定
```

## 三、第二步：从生效范围推导投递通道

确立了"什么知识在什么场景需要"之后：**"怎么让它在那个场景下正好出现？"**

### 3.1 🌐 全局常驻型 → 磁盘写入通道

这类知识（Rule、Convention、Architecture）**必须 100% 送达**，大模型不能跳过。

唯一能保证 100% 送达的方式：**写入磁盘静态文件**，利用 IDE 自带的 System Prompt 注入机制强制加载。

- 无论大模型是否"愿意"调用 MCP，静态文件中的内容都会在每次对话中自动注入
- 大模型没有"选择读或不读"的权利

→ 全局常驻型走**磁盘写入通道**

### 3.2 📦 模块绑定型 + 🔗 意图触发型 → MCP 动态查询通道

这类知识（API Spec、Schema、Skill、Workflow、Glossary、Reference）数量庞大、专业性强。

如果全量写入静态文件，Token 预算瞬间耗尽。它们需要的不是"始终在"，而是**"在正确的时间精确出现"**。

动态按需检索是唯一可行的方案：由 Local Daemon 作为 MCP Server，AI 在需要时主动查询。

→ 模块绑定型 + 意图触发型走**MCP 动态查询通道**

## 四、第三步：通道内细分 → L1 / L2 / L3 三层体系

两条通道（磁盘写入 / MCP 查询）已经确定。但磁盘写入通道内的知识还有两种不同性质：

### 4.1 L1 元知识：SkyBrain 自身的"使用说明书"

磁盘写入通道中，除了从内容形态推导出的全局常驻知识（Rule、Convention、Architecture），还有一类特殊知识：

> **SkyBrain 系统自身的元知识** —— 告诉 AI "如何正确使用 SkyBrain"

例如："写 API 前必须先通过 MCP 查询 SkyBrain 知识库"、"遇到重构任务请先调用 SkyBrain 的 architecture skill"。

这类知识的特点：
- 不是任何内容形态分类中的一员
- 是 SkyBrain 系统内置的，跨项目通用
- 变更频率极低（仅跟随 Daemon 版本升级）
- 设计目的单一：**驱动 AI 去查询 L3**

→ 独立为 **L1 元知识**

### 4.2 L2 固化知识：全局常驻的项目知识

从内容形态推导出的全局常驻型知识（Rule、Convention、Architecture）：
- 项目专属
- 由团队 TL 管理
- 变更频率中等（随团队规范演进）

→ 归类为 **L2 固化知识**

### 4.3 L3 动态扩展知识：按需检索的长尾知识

MCP 通道内的所有知识（模块绑定型 + 意图触发型）：
- 数量庞大，随项目持续增长
- 只在特定场景下需要
- 通过 MCP 按需查询

→ 归类为 **L3 动态扩展知识**

## 五、完整推导链路总结

```
内容形态                         生效范围              投递通道         分级
─────────────────────────────────────────────────────────────────────────────
Rule           (约束型)        ──→ 🌐 全局常驻 ──→ 磁盘写入 ──→  L2
Convention     (约束型)        ──→ 🌐 全局常驻 ──→ 磁盘写入 ──→  L2
Architecture   (认知型)        ──→ 🌐 全局常驻 ──→ 磁盘写入 ──→  L2
Glossary       (认知型)        ──→ 📦 模块绑定 ──→ MCP查询 ──→  L3
API Spec       (契约型)        ──→ 📦 模块绑定 ──→ MCP查询 ──→  L3
Schema         (契约型)        ──→ 📦 模块绑定 ──→ MCP查询 ──→  L3
Reference      (操作型)        ──→ 📦 模块绑定 ──→ MCP查询 ──→  L3
Skill          (操作型)        ──→ 🔗 意图触发 ──→ MCP查询 ──→  L3
Workflow       (操作型)        ──→ 🔗 意图触发 ──→ MCP查询 ──→  L3

+ L1 元知识（系统内置，不属于任何内容形态）
  ──→ SkyBrain 自身使用说明书 ──→ 磁盘写入 ──→ L1
```

### L1 与 L2 的关键区别

| 维度 | L1 元知识 | L2 固化知识 |
|------|----------|------------|
| 来源 | SkyBrain 系统内置 | 团队录入 + 冷启动扫描 |
| 变更频率 | 极低（跟随 Daemon 版本升级） | 中等（团队规范演进） |
| 跨项目通用 | ✅ 是 | ❌ 否 |
| 设计目的 | 驱动 AI 去查询 L3 | 告诉 AI 项目的基本常识 |

## 六、方案面临的核心挑战与应对策略

### 挑战 1：知识的生效范围并非完全固定

某条知识可能同时适配多个生效范围。比如一个支付鉴权 Skill 既属于操作型（意图触发），又跟 order 模块强绑定（模块触发）。

**应对**：采用"默认值 + 手动覆盖"策略 ——
- 每种知识类型在创建时自动赋予默认生效范围
- TL 或创建者可以在知识属性中手动覆盖
- 知识点可以同时打上多个生效标签

### 挑战 2：模块绑定的粒度问题

多 Module 的代码库目录结构千差万别，"模块"的边界如何定义？

**应对**：
- 支持通过项目级 `skybrain.yaml` 显式声明模块路径映射
- 若无声明，默认按顶级目录作为隐式模块划分
- 知识创建时可以由 AI 自动推断关联的文件 glob pattern

### 挑战 3：意图触发的匹配精度

AI 说"写一个支付接口"，SkyBrain 需要识别"支付"意图并拉取相关 Skill。如何区分"支付"和"退款"？

**应对**：
- 知识入库时生成语义向量，与 AI 当前任务描述做相似度匹配
- 每条知识维护一个"触发短语"列表，提升匹配精度
- 支持多级意图标签体系（一级：支付 → 二级：鉴权/回调/退款）

---

## 七、结论

本次脑洞确立了从内容形态到分级加载体系的**完整推导链路**：

> **第一步**：从四大内容形态（操作型/契约型/认知型/约束型）的特征出发，推导出每种知识的天然生效范围（全局常驻/模块绑定/意图触发）。
>
> **第二步**：从生效范围推导投递通道 —— 全局常驻型必须走磁盘写入，确保 100% 送达；模块绑定型和意图触发型必须走 MCP 动态查询，实现按需精准投递。
>
> **第三步**：通道内进一步细分为 L1（元知识，系统内置）、L2（固化知识，全局常驻项目知识）、L3（动态扩展知识，按需检索）。
>
> 整个推导过程**从内容形态出发，而非从技术方案出发**，确保了 L1/L2/L3 分级加载体系有坚实的业务逻辑支撑。

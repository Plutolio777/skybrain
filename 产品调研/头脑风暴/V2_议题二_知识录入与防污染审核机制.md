# SkyBrain 解决方案 V2 — 议题二：极简知识录入与防污染审核机制

> 状态：已完成讨论
> 阶段：V2 解决方案与产品雏形
> 核心议题：如何让开发者心甘情愿地录入知识？同时如何防止低质量的个人知识污染中央大脑？

---

## 一、 知识录入的交互演进：放弃重型插件，拥抱 MCP 反向写入

在探讨开发者如何上传隐性知识时，我们果断放弃了传统的“强绑定 IDE 插件（如右键菜单上传）”的路线，原因在于主流 AI IDE 繁多（Cursor, Trae, Copilot, Cline 等），插件适配成本极高且极易过时。

我们确立了基于 **MCP（Model Context Protocol）的“对话式反向写入”** 为主要录入交互方案。

*   **交互形态**：开发者完全不需要离开写代码的上下文，也无需点击任何按钮。只需在 AI 聊天的对话框中输入：“把这段重试逻辑作为开发规范，保存到 SkyBrain”。
*   **底层机制**：SkyBrain 本地客户端通过 MCP 向 IDE 暴露 `save_to_skybrain(content, tags)` 这样的“写入型”工具（Tool）。大模型理解了开发者的意图后，自动调用该工具，将梳理好的规则内容反向推送给本地客户端。
*   **价值**：极致的“无摩擦”体验。开发者在解决 bug、重构代码的自然流中，顺嘴说一句就能完成知识沉淀。

---

## 二、 核心痛点解决：防污染的“知识三级生命周期”

“极简录入”必然带来“垃圾输入”。如果任何开发者通过 MCP 喊一嗓子就能把规则写入“中央大脑”，团队的全局上下文将迅速被错误信息和互相冲突的规则污染。

为此，SkyBrain 设计了类似 Git PR（Pull Request）流的 **“知识三级生命周期隔离与审核机制”**：

### 1. 个人沙盒层 (Personal Sandbox) —— "隔离验证区"
*   **机制**：所有通过 MCP `save_to_skybrain` 写入的知识，默认进入开发者的本地沙盒。
*   **物理隔离**：这部分知识被本地的 SkyBrain MCP Server 存储在开发者本机的本地数据库中，绝不上传云端。
*   **可见性**：仅对自己可见。第二天开发者提问时，本地 MCP Server 会合并检索“云端大脑”和“本地沙盒”，开发者能享受到自己沉淀的知识，但不会影响任何人。

### 2. 团队草稿区与提案 (Team Proposal) —— "云端待审核"
*   **流转机制**：当开发者在个人沙盒中验证某条规则（如《安全鉴权范例》）非常有效后，可以通过 MCP 指令：“把这条安全鉴权规则提报给团队”。
*   **物理流转**：本地 MCP Server 此时才将该数据打包，通过 API 上传至云端的 Web 控制台。
*   **AI 预审**：进入云端后，SkyBrain 后台的大模型会进行“冲突检测”。如果该提案与现有的全局规范存在硬冲突，直接打回给提交者；若无硬伤，则进入 `Proposal`（提案）状态等待人工审核。

### 3. 中央大脑 (Global Brain) —— "全局真理"
*   **审核机制 (Knowledge Review)**：团队的技术 TL 或架构师登录 Web 端控制台（实体界面），在“待审核面板”中看到小 A 提交的提案。
*   **生效范围**：TL 确认无误并点击 【Approve】 后，该知识正式并入“中央大脑”。此时，该知识对全团队成员的 MCP Server 可见，全员生效。

---

## 三、 架构推论：本地部署的必然性

这一套“沙盒隔离 + MCP交互”的防污染机制，从逻辑上推导出了一个必然的架构要求：**SkyBrain 必须在每个开发者的电脑上部署一个轻量级的本地端（Daemon/客户端）。**

只有当 MCP Server 运行在本地时：
1. 才能拥有毫秒级的读写速度，保障“个人沙盒”的极致体验。
2. 才能读取和感知本地文件系统的绝对路径，以便提取特定代码。
3. 才能充当“数据守门员”，将庞杂的个人实验数据截留在本地，仅将提炼后的高价值提案（Proposal）放行至云端 Web 实体。

---

## 四、 结论

议题二不仅解决了一线开发者的录入摩擦力问题，更重要的是建立了一套“去中心化生产 + 集中式把控”的护城河。通过 **“MCP 写入 -> 个人本地沙盒 -> AI 预审 -> Web 端 TL 审批 -> 中央大脑”** 的闭环，SkyBrain 实现了在保障知识库纯洁性的前提下，最大化地榨取团队的群体智慧。
